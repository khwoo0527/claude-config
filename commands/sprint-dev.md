sprint{n}.md에 따라 스프린트를 구현하는 오케스트레이터.

## 인수

`$ARGUMENTS`: 스프린트 번호 (예: `34`, `@sprint35`)
- 숫자만 추출하여 `docs/sprint/sprint{n}.md` 경로를 결정한다.

## 실행 절차

### 1단계: 프로젝트 컨텍스트 로드

1. `/CLAUDE.md`를 읽어 기술 스택, 프로젝트 구조, 빌드 명령, 코드 컨벤션을 확인한다.
2. `.claude/rules/` 하위에서 CLAUDE.md의 기술 스택에 해당하는 규칙 파일을 **모두** 로드한다.
   - 해당 기술의 네이밍 컨벤션, 구조 규칙, 보안 규칙, 성능 가이드라인을 **완전히 숙지**한다.
   - 이 규칙들은 구현, 검증, 코드 정리의 **전문 기준**이 된다.
   - **해당 기술 스택의 rules/tech/{tech}.md가 없는 경우**: 작업을 시작하기 전에 사용자에게 rules/tech/{tech}.md 생성을 먼저 제안한다. 사용자가 동의하면 **`.claude/templates/RULES-TEMPLATE.md`를 참조하여** 해당 기술 스택의 시니어 수준 규칙 파일(언어 + 프레임워크 + 실전 패턴)을 생성한 뒤 작업을 진행한다.
3. `.claude/rules/workflow/sprint-workflow.md`를 읽어 워크플로우 규칙을 확인한다.

### 2단계: 스프린트 문서 읽기

1. `docs/sprint/sprint{n}.md`를 **전체** 읽는다.
2. 다음을 파악한다:
   - **실행 플랜**: Phase 구조와 각 Phase의 Task 목록
   - **의존성 그래프**: Task 간 선후 관계
   - **병렬 가능 표시**: Phase 제목에 `(병렬 가능)` 또는 본문에 병렬 관련 안내가 있는지
   - **각 Task의 skill 헤더**: 참조할 스킬
   - **각 Task의 Step**: 구체적 실행 순서
   - **각 Task의 검증 명령과 커밋 메시지**

### 3단계: 브랜치 준비

1. 현재 브랜치가 `sprint{n}`이 아니면:
   - `develop` 브랜치에서 `git checkout -b sprint{n}` 으로 생성
2. 이미 `sprint{n}` 브랜치면 그대로 진행

### 4단계: Phase 순차 실행

실행 플랜의 Phase를 **순서대로** 처리한다. 각 Phase에 대해:

#### 병렬 Phase인 경우 (Phase 제목에 "병렬 가능" 표시)

1. Skill 도구로 `dispatching-parallel-agents`를 로드하여 병렬 구성 가이드를 참조한다.
2. **사용자에게 팀 구성 제안**:
   ```
   Phase {N} — 병렬 실행 가능

   | Agent | Task | skill | 설명 |
   |-------|------|-------|------|
   | agent-1 | Task X | systematic-debugging | ... |
   | agent-2 | Task Y | feature-dev | ... |

   팀으로 병렬 실행할까요? (y/순차로 진행)
   ```
3. 사용자가 승인하면:
   - Agent 도구로 **동시에** 여러 에이전트를 실행한다.
   - 각 에이전트의 프롬프트에는 해당 Task의 전체 Step, 파일 목록, 검증 명령을 포함한다.
   - **기술 스택 규칙 주입**: CLAUDE.md에서 확인한 기술 스택과 rules/tech/{tech}.md의 핵심 규칙을 에이전트 프롬프트에 포함한다.
   - **skill 주입**: Task의 `skill:` 헤더에 명시된 스킬명과 핵심 원칙을 에이전트 프롬프트에 포함한다.
   - 에이전트에게 **커밋은 하지 말 것**을 명시한다 (커밋은 메인 오케스트레이터가 수행).
   - 에이전트 완료 후 메인 세션에서 **simplify + 커밋**을 순차 수행한다.
4. 사용자가 거부하면 → 순차 Phase와 동일하게 처리

#### 순차 Phase인 경우

Phase 내 각 Task를 순서대로 실행한다. **매 Task마다 아래 체크리스트를 반드시 수행한다:**

```
Task 실행 체크리스트 (건너뛰기 금지):

[코딩 전]
1. rules 숙지 — sprint{n}.md의 "코드 컨벤션" 섹션 + rules/tech/{tech}.md 핵심 규칙을 다시 읽는다
2. skill 로드 — `skill:` 헤더가 있으면 Skill 도구 호출

[코딩 중]
3. Step 실행 — sprint{n}.md의 Step 순서대로, rules를 내재화한 상태에서 코딩
   - 시니어 개발자가 자연스럽게 따르는 수준으로 코드 컨벤션 적용
   - 하드코딩, 네이밍, 타입 안전성, 컴포넌트 설계, import 순서 등 전부 포함
   - "나중에 고치지" 금지 — 처음부터 rules 수준으로 작성

[코딩 후]
4. rules 준수 재검증 — 작성한 코드를 rules/tech/{tech}.md 원본과 대조하여 위반 사항 수정
5. 빌드 검증 — CLAUDE.md에 정의된 빌드 명령 실행
6. 검증 통과 — 명시된 검증 명령 실행
7. simplify — Skill("simplify") 실행 (생략 불가)
8. 커밋 — 명시된 커밋 메시지로
```

각 단계 상세:

1. **Task 시작 알림**: `Task {N}: {제목} 시작`
2. **skill 로드**: Task의 `skill:` 헤더에 명시된 스킬이 있으면 Skill 도구로 로드하고 skill별 실행 전략에 따라 진행한다.
3. **Step 실행**: sprint{n}.md에 명시된 Step을 순서대로 수행한다.
   - **기술 스택 규칙 적용**: 구현 시 rules/tech/{tech}.md의 베스트 프랙티스를 적극 반영한다.
   - 해당 기술의 관용적 패턴(idiomatic patterns)을 사용하여 최고 품질의 코드를 작성한다.
4. **빌드 검증**: CLAUDE.md에 정의된 빌드 명령으로 빌드가 성공하는지 확인한다. 실패 시 원인 분석 후 수정.
5. **추가 검증**: sprint{n}.md에 명시된 검증 명령을 실행하고 결과를 확인한다.
6. **simplify**: Skill 도구로 `simplify`를 로드하고, 이번 Task에서 수정한 코드를 정리한다.
7. **커밋**: sprint{n}.md에 명시된 커밋 메시지로 커밋한다. **반드시 사용자 확인 후 커밋**.
8. **완료 보고**: 완료 기준 체크리스트를 표시한다.

### 5단계: Phase 체크포인트

각 Phase 완료 후 사용자에게 보고하고 다음 Phase 진행 여부를 확인한다:

```
Phase {N} 완료

완료된 Task:
- [x] Task X: {제목} — {검증 결과 요약}
- [x] Task Y: {제목} — {검증 결과 요약}

다음: Phase {N+1} — {Task 목록 요약}
계속 진행할까요?
```

**예외**: 마지막 Phase가 아닌 경우에만 확인을 묻는다. 사용자가 "끝까지" 또는 "전부" 같은 지시를 한 경우 체크포인트를 건너뛴다.

### 6단계: 최종 검증

모든 Phase 완료 후:

1. Skill 도구로 `verification-before-completion`을 로드한다.
2. sprint{n}.md의 **최종 검증 계획** 섹션에 명시된 검증을 모두 실행한다.
3. 검증 명령 실행 결과를 **실증(로그/출력)**으로 제시한다. 결과 없이 "통과했다"고 주장하지 않는다.
4. **사용자 테스트 가능 상태 확인**: 빌드 오류 없음 + 앱 실행됨 + 화면에서 직접 확인 가능한 상태인지 점검한다.
5. 결과를 요약 보고한다:
   ```
   Sprint {N} 구현 완료

   | 검증 항목 | 결과 |
   |-----------|------|
   | 빌드 | {결과} |
   | 린트/포맷 | {결과} |
   | 테스트 | {결과} |
   | 실행 테스트 | {결과} |
   | 사용자 테스트 가능 | ✅ 화면 확인 + 동작 테스트 가능 |
   | ... | ... |

   사용자 리뷰를 진행해주세요:
   1. 화면을 직접 확인하고 동작을 테스트해주세요
   2. 수정할 것이 있으면 알려주세요
   3. "확인 완료" 후 → `sprint-close` → `sprint-review`
   ```
6. **사용자가 "확인 완료"할 때까지 다음 Sprint로 넘어가지 않는다.**

## 검증 명령

### 필수 검증 (매 Task)
```bash
# CLAUDE.md에 정의된 빌드 명령 실행
# 빌드 결과 확인: 성공 + 에러 없음 필수
# Warning은 sprint{n}.md 기준으로 허용 범위 판단

# CLAUDE.md에 린트/포맷 명령이 있으면 실행
# CLAUDE.md에 테스트 명령이 있으면 실행
```

### 코드 품질 체크리스트 (매 Task)

**반드시 rules/tech/{tech}.md 원본을 읽고** 아래를 점검한다. sprint{n}.md의 "코드 컨벤션" 섹션은 핵심 발췌일 뿐, **원본 rules가 진짜 기준**이다.

```
[아키텍처]
- [ ] 파일이 올바른 폴더에 위치하는가 (app/ = 라우팅만, features/ = 비즈니스 로직)
- [ ] 의존 방향이 올바른가 (app → features → shared → services)

[하드코딩 금지]
- [ ] 매직 넘버/문자열이 없는가 → 상수 파일(constants/)에 정의
- [ ] 색상/사이즈 하드코딩이 없는가 → 테마 토큰/NativeWind 클래스 사용
- [ ] URL/키 하드코딩이 없는가 → 환경 변수 사용

[타입 안전성]
- [ ] any 사용이 없는가
- [ ] as 타입 단언을 남용하지 않았는가
- [ ] ! non-null assertion이 없는가

[컴포넌트 품질]
- [ ] Props가 interface로 타입 정의되었는가
- [ ] 100줄 초과 컴포넌트가 없는가
- [ ] 인라인 style/onPress가 없는가 (NativeWind + useCallback)
- [ ] 네이밍 컨벤션을 따르는가 (rules/tech/{tech}.md 기준)

[에러/보안]
- [ ] 에러 처리가 적절한가 (catch unknown, 삼키기 금지)
- [ ] 보안 취약점이 없는가 (하드코딩 시크릿, 입력 미검증)

[성능]
- [ ] 불필요한 리렌더/객체 생성이 없는가
- [ ] 리스트에 가상화가 적용되었는가 (10개+ 아이템)
```

### 데이터 호환성 검증 (데이터 모델 변경 시)
```
1. 기존 데이터로 로드 테스트
2. 로드 후 다시 저장 → 재로드 확인
3. 새 필드에 기본값이 정상 적용되는지 확인
```

## skill별 실행 전략

Task의 `skill:` 헤더에 따라 Step 실행 방식이 달라진다:

| skill | 실행 방식 |
|-------|----------|
| `systematic-debugging` | Skill 로드 → 원인 분석 → 수정 |
| `feature-dev:feature-dev` | code-explorer Agent로 탐색 → 결과 기반 구현 |
| `brainstorming` | Skill 로드 → 설계 탐색 → 사용자 확인 → 구현 |
| `simplify` | Skill 로드 → 기능 변경 없이 코드 정리 |
| (없음) | Step을 그대로 실행 |

## 에러 복구 전략

### 빌드 실패 시
1. 빌드 에러 메시지에서 파일:줄번호 추출
2. 해당 파일을 읽어 컨텍스트 파악
3. rules/tech/{tech}.md의 빌드 관련 규칙을 참조하여 에러 유형별 대응
4. 수정 후 재빌드, 3회 실패 시 사용자에게 보고

### 런타임 에러 시
1. 에러 타입과 스택트레이스 분석
2. 기술 스택별 흔한 런타임 에러 패턴 확인 (rules/tech/{tech}.md 참조)
3. 3회 시도 후 해결 불가 시 사용자에게 보고

### UI/프론트엔드 관련 이슈 시 (해당 시)
1. 스타일/레이아웃 문제 → 기존 컴포넌트와 비교
2. 이벤트/상태 관리 문제 → 프로젝트 패턴 참조
3. 프레임워크 특화 이슈 → rules/tech/{tech}.md 안티패턴 확인

## 주의사항

- **sprint{n}.md가 Single Source of Truth**: 문서에 명시되지 않은 작업은 하지 않는다.
- **계획과 다른 결정이 필요하면 사용자에게 확인**한다.
- **커밋 메시지**: sprint{n}.md에 명시된 것을 그대로 사용한다.
- **커밋 전 반드시 사용자 확인**: 임의로 커밋/푸시하지 않는다.
- **검증 실패 시**: 실패 원인을 분석하고 수정한 뒤 재검증한다. 3회 실패 시 사용자에게 보고한다.
- **코드 리뷰는 이 커맨드의 범위 밖**: `sprint-review` 에이전트가 담당한다.
- **worktree 사용 금지**: `git checkout -b`로 브랜치를 생성한다.
- **기술 스택 규칙 준수**: rules/tech/{tech}.md의 베스트 프랙티스를 반드시 적용한다. 범용적인 코드가 아닌 해당 기술의 **관용적이고 전문적인** 코드를 작성한다.
- **패키지 관리**: CLAUDE.md에 정의된 패키지 관리 방식과 호환성 기준 준수.
