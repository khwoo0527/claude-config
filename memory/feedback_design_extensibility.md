description: 파일명 하드코딩 같은 짧은 시야 금지. 동적/위임/패턴 기반 설계 가능한지 먼저 검토. 페이퍼워크 늘리는 설계는 안티패턴
type: feedback
---

규칙: 문서·커맨드·시스템을 설계할 때 다음을 **사전에** 고려하고 결정한다.

1. **책임 분리**: 어떤 모듈/문서가 어떤 책임을 갖는지 명확히. 한 파일이 여러 책임을 갖거나, 여러 파일이 같은 정보를 중복 보유하지 않도록.
2. **확장성**: 항목이 추가/변경/삭제될 가능성이 있는가? 있다면 동적 방식(글롭, 패턴 매칭, 위임) 우선 검토.
3. **미래 정책 변경**: 정책(로딩 방식, 우선순위, 분류 등)이 바뀔 가능성이 있는가? 있다면 정책을 **한 곳**에 정의하고 다른 곳은 그 정의를 따르도록 위임 구조로 설계.
4. **하드코딩은 최후의 수단**: 파일명·고정값 박기 전에 "이게 늘어나면 어디를 수정해야 하나?" 자문. 두 곳 이상 수정 필요하면 설계 재검토.

**Why:**
- 짧은 시야로 파일명 박아두면 항목 추가 때마다 문서 동기화 페이퍼워크 폭증 → 누락 위험 + 사용자 시간 낭비.
- 책임이 흩어지면 변경 영향 범위가 커지고, 어디를 손대야 할지 매번 헷갈림.
- 사전에 5분 더 고민하면 수백 번의 부수 작업이 사라짐.

**How to apply:**
- 새 문서/커맨드/규칙 작성 시작 전에 이 4가지를 **명시적으로 검토**하고 답변 가능해야 함:
  1. 이 문서의 책임은 무엇이고, 다른 문서와 어떻게 분리되는가?
  2. 항목이 추가/변경될 가능성이 있는가? 있다면 동적으로 처리하는가?
  3. 정책이 바뀌면 몇 곳을 수정해야 하는가? (이상적으로는 1곳)
  4. 하드코딩하려는 값이 정말 변하지 않는 상수인가?
- 하드코딩하려는 순간 **잠깐 멈추고** 글롭/위임/패턴으로 해결 가능한지 먼저 검토.
- "예시" 라는 명목으로 파일명 나열하는 것도 동기화 부담 → 정말 필요한지 재검토 (보통 불필요).
- 사용자가 같은 지적("왜 매번 문서 수정해야 하나?")을 반복하지 않도록 능동 적용.
