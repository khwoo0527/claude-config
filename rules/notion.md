# Notion 문서 작성 규칙

> 이 파일은 AI가 프로젝트 정보를 Notion에 정리할 때 따르는 **문서 작성 품질 가이드**입니다.
> API 코드가 아닌, **어떻게 쓰면 보기 좋고 내용이 풍부한 Notion 문서가 되는지**에 집중합니다.
> 프로젝트별 Notion 페이지 ID, 연동 설정은 `/CLAUDE.md`에 정의합니다.

---

## 핵심 원칙

1. **스캔 가능성 (Scannability)** — 3초 안에 페이지의 전체 구조를 파악할 수 있어야 한다. 위에서 아래로 훑으면 핵심이 잡혀야 한다
2. **적정 밀도** — 빈 페이지도, 텍스트 벽도 안 된다. 모든 문장이 존재 이유가 있어야 하고, 시각 요소(표/콜아웃/토글)로 밀도를 조절한다
3. **블록 다양성** — 텍스트만으로 채우지 않는다. Notion의 강점은 블록 타입의 다양성. 표, 콜아웃, 토글, 코드 블록을 적극 활용한다
4. **계층적 구조** — H1 → H2 → H3 → 본문의 계층을 지킨다. 건너뛰기(H1 바로 뒤에 H3) 금지
5. **자기 완결성** — 그 페이지만 읽으면 해당 주제를 이해할 수 있어야 한다. "다른 데 가서 봐라"를 최소화

## 블록 선택 가이드

> Notion의 핵심은 **상황에 맞는 블록을 고르는 것**이다.
> 같은 정보도 어떤 블록으로 표현하느냐에 따라 가독성이 완전히 달라진다.

### 언제 어떤 블록을 쓰는가

| 상황 | 권장 블록 | 이유 |
|------|----------|------|
| 2개 이상 항목 비교 | **표 (Table)** | 나란히 놓아야 차이가 보인다 |
| 속성-값 쌍 나열 (설정값, 환경 정보 등) | **표 (Table)** | 목록보다 정렬이 깔끔 |
| 주의사항, 경고, 중요 알림 | **콜아웃 (Callout)** | 시각적으로 눈에 띄어야 하는 정보 |
| 상세 내용이 길지만 항상 볼 필요 없음 | **토글 (Toggle)** | 접어두면 페이지가 깔끔, 필요할 때 펼침 |
| 순서가 있는 절차/단계 | **번호 목록** | 순서가 중요한 정보는 번호로 |
| 순서 없는 특징/항목 나열 | **글머리 기호 목록** | 3~7개가 적정. 10개 넘으면 그룹핑 필요 |
| 코드, 명령어, 설정값 | **코드 블록** | 인라인 코드(`)는 짧은 것만, 여러 줄이면 코드 블록 |
| 섹션 간 시각적 분리 | **구분선 (Divider)** | 제목만으로 분리가 약할 때 보조적으로 사용 |
| 완료/미완료 추적이 필요한 항목 | **할 일 목록 (To-do)** | 체크박스로 진행 상태 관리 |
| 핵심 수치/상태 요약 | **콜아웃 + 볼드** | 페이지 상단에 배치하여 한눈에 현황 파악 |

### Bad vs Good 예시

**정보: 서버 3대의 환경 정보**

```
❌ Bad — 글머리 기호로 나열
• 개발 서버: 192.168.1.10, Ubuntu 22.04, 8GB RAM
• 스테이징 서버: 192.168.1.20, Ubuntu 22.04, 16GB RAM
• 운영 서버: 10.0.1.100, Ubuntu 22.04, 32GB RAM
```

```
✅ Good — 표로 비교
| 환경 | IP | OS | RAM |
|------|-----|------|-----|
| 개발 | 192.168.1.10 | Ubuntu 22.04 | 8GB |
| 스테이징 | 192.168.1.20 | Ubuntu 22.04 | 16GB |
| 운영 | 10.0.1.100 | Ubuntu 22.04 | 32GB |
```

**정보: 배포 전 확인 사항**

```
❌ Bad — 글머리 기호 (순서가 있는 작업인데 순서 없는 목록)
• 테스트 통과 확인
• 빌드 성공 확인
• 스테이징 배포
• QA 검증
• 운영 배포
```

```
✅ Good — 번호 목록 + 할 일 목록 (상태 추적도 가능)
1. [ ] 테스트 전체 통과 확인
2. [ ] 빌드 성공 확인
3. [ ] 스테이징 배포 및 검증
4. [ ] QA 승인
5. [ ] 운영 배포
```

**정보: 중요한 제약 사항**

```
❌ Bad — 일반 텍스트에 묻혀서 놓치기 쉬움
참고로 이 API는 1분에 60회까지만 호출할 수 있습니다. 초과하면 429 에러가 발생합니다.
```

```
✅ Good — 콜아웃으로 시각적 강조
⚠️ Rate Limit: 1분 60회. 초과 시 429 에러 → 배치 처리 또는 큐 사용 필요
```

## 페이지 구조 설계

### 기본 레이아웃 패턴

모든 Notion 페이지는 이 순서를 따른다:

```
📄 페이지 제목
├── [콜아웃] 한줄 요약 또는 현재 상태     ← 3초 안에 핵심 파악
├── [구분선]
├── H2: 섹션 1                          ← 논리적 그룹
│   ├── 본문 (텍스트/표/목록)
│   └── [토글] 상세 내용 (길면 접기)
├── H2: 섹션 2
│   └── ...
└── H2: 참고/관련 링크                    ← 맨 아래에 부가 정보
```

### 페이지 상단 — 3초 요약

페이지를 열었을 때 **스크롤 없이** 보이는 영역에 핵심 정보를 배치한다:

```
✅ Good — 상단에 현황 콜아웃
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📌 프로젝트 현황
  상태: 개발 중 | 버전: v2.3.0 | 다음 릴리즈: 2026-04-15
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
(이하 상세 내용)
```

```
❌ Bad — 상단에 목적/배경 장문
이 프로젝트는 2025년 3월에 시작되었으며,
기존 시스템의 문제점을 해결하기 위해...
(스크롤해야 현재 상태를 알 수 있음)
```

### 계층 규칙

| 수준 | 용도 | 예시 |
|------|------|------|
| H1 | 페이지 제목 (1개만) | 프로젝트에서 자동 설정됨 |
| H2 | 대 섹션 구분 | "시스템 구성", "API 명세", "릴리즈 이력" |
| H3 | 소 섹션 구분 | "인증 API", "조회 API", "등록 API" |
| **볼드 텍스트** | 항목 내 강조 | 표 헤더, 핵심 키워드 |
| 일반 텍스트 | 설명 본문 | — |

> **금지**: H1을 여러 번 사용, H2 없이 바로 H3 사용, 볼드를 제목 대신 사용

## 콘텐츠 작성 원칙

### 적정 밀도 — "있어야 할 건 다, 없어도 될 건 없이"

**포함해야 할 것:**
- 의사결정에 필요한 정보 (왜 이렇게 했는지, 대안은 뭐였는지)
- 현재 상태와 다음 단계
- 재현/실행에 필요한 구체적 값 (버전, 경로, 명령어)
- 주의사항과 제약 조건

**제외해야 할 것:**
- 누구나 아는 일반론 ("테스트는 중요합니다")
- 중복 설명 (같은 내용을 다른 섹션에서 반복)
- 빈 섹션 (내용이 없으면 섹션 자체를 제거)
- 과도한 수식어 ("매우 중요한", "반드시 꼭")

### Good vs Bad 비교

```
❌ Bad — 일반론 + 수식어 남발
## 테스트
테스트는 소프트웨어 품질을 보장하기 위해 매우 중요한 활동입니다.
반드시 꼼꼼하게 테스트를 수행해야 합니다.
단위 테스트, 통합 테스트, E2E 테스트 등 다양한 종류가 있습니다.
```

```
✅ Good — 이 프로젝트에서 필요한 구체적 정보만
## 테스트
| 종류 | 범위 | 실행 명령 | 기준 |
|------|------|----------|------|
| 단위 | 핵심 비즈니스 로직 | `npm test` | 커버리지 80% 이상 |
| 통합 | API 엔드포인트 | `npm run test:integration` | 전체 통과 |
| E2E | 주요 사용자 시나리오 3개 | `npm run test:e2e` | 전체 통과 |

⚠️ PR 머지 전: 단위 + 통합 필수. E2E는 릴리즈 전에만.
```

### 길이 판단 기준

| 문서 유형 | 적정 길이 | 과한 경우 |
|----------|----------|----------|
| 프로젝트 개요 | A4 1~2장 분량 | 배경 설명이 본문의 절반 이상 |
| 기능 명세 (개별) | 토글 1개에 들어갈 분량 | 스크롤 3번 이상 필요 → 하위 페이지로 분리 |
| 릴리즈 노트 (개별) | 변경사항 목록 + 주의사항 | 코드 diff까지 포함 → 코드는 링크로 |
| 아키텍처 | 다이어그램 + 컴포넌트별 설명 | 코드 레벨 상세 → 코드 문서로 분리 |
| 회의록/결정 기록 | 결론 + 근거 + 액션 아이템 | 논의 과정 전체 기록 → 결론 위주로 정리 |

## 시각적 디자인 원칙

### 아이콘 활용

페이지와 콜아웃에 아이콘을 적절히 사용하면 스캔 속도가 올라간다:

| 용도 | 권장 아이콘 | 예시 |
|------|-----------|------|
| 프로젝트/제품 | 🏗️ 📦 🚀 | 프로젝트 페이지 아이콘 |
| 경고/주의 | ⚠️ 🔴 | 콜아웃: "운영 DB 직접 수정 금지" |
| 정보/팁 | 💡 ℹ️ | 콜아웃: "디버깅 시 유용한 쿼리" |
| 성공/완료 | ✅ 🟢 | 콜아웃: "v2.3.0 배포 완료" |
| 진행 중 | 🔄 🟡 | 콜아웃: "QA 진행 중" |
| 문서/명세 | 📄 📋 | 하위 페이지 아이콘 |
| 설정/환경 | ⚙️ 🔧 | 환경 설정 페이지 |

> **과용 금지**: 모든 글머리 기호에 이모지를 붙이면 오히려 시각적 노이즈. 페이지 아이콘과 콜아웃에만 사용하는 것을 권장.

### 여백과 간격

- **섹션(H2) 사이**: 구분선 또는 빈 줄 1개로 시각적 분리
- **관련 블록 사이**: 붙여서 배치 (불필요한 빈 줄 넣지 않음)
- **표 앞뒤**: 빈 줄 1개씩 (표가 본문에 붙으면 답답해 보임)
- **콜아웃 앞뒤**: 빈 줄 1개씩

### 색상 (콜아웃 배경)

| 색상 | 용도 |
|------|------|
| 빨간 배경 | 금지, 위험, 장애 |
| 노란 배경 | 주의, 진행 중, 임시 |
| 파란 배경 | 정보, 참고, 팁 |
| 초록 배경 | 완료, 성공, 승인 |
| 회색 배경 | 부가 설명, 덜 중요한 참고 |

## 토글 활용 전략

토글은 Notion의 강력한 무기이다. **페이지를 짧게 유지하면서도 정보를 풍부하게** 담을 수 있다.

### 언제 토글을 쓰는가

| 상황 | 토글 사용 | 이유 |
|------|----------|------|
| 상세 설명이 5줄 이상 | ✅ 사용 | 메인 흐름이 끊기지 않도록 접기 |
| 예시/샘플 코드 | ✅ 사용 | 코드가 길면 페이지가 길어짐 |
| FAQ/자주 묻는 질문 | ✅ 사용 | 질문을 토글 제목으로, 답을 내용으로 |
| 이전 버전 기록 | ✅ 사용 | 현재 버전이 메인, 이전은 접어두기 |
| 핵심 정보 (항상 봐야 할 것) | ❌ 미사용 | 접혀 있으면 놓칠 수 있음 |
| 1~2줄 짧은 설명 | ❌ 미사용 | 클릭 한 번이 오히려 비효율 |

### 토글 제목 작성법

```
❌ Bad — 제목이 모호
▶ 상세 내용
▶ 자세히 보기
▶ 기타

✅ Good — 제목만 봐도 내용을 예측 가능
▶ API 응답 JSON 예시 (200 OK)
▶ v2.2.0 → v2.3.0 마이그레이션 절차
▶ 에러 코드별 대응 방법
```

## 표 작성 가이드

표는 Notion에서 **가장 정보 밀도가 높은 블록**이다. 잘 쓰면 10줄 텍스트를 3줄 표로 압축할 수 있다.

### 표 사용 원칙

- **열은 2~5개**가 적정. 6개 이상이면 모바일에서 깨지고 가독성 저하
- **첫 번째 열은 식별자** 역할 (이름, 항목명, 키)
- **마지막 열은 가장 긴 텍스트** 배치 (비고, 설명 등)
- **셀 내용은 짧게** — 셀 안에 3줄 이상 들어가면 표가 아니라 본문으로 빼야 함

### 자주 쓰는 표 패턴

**비교 표** — 옵션/대안 비교 시:
```
| 구분 | 옵션 A | 옵션 B | 선택 |
|------|--------|--------|------|
| 속도 | 빠름 | 보통 | A |
| 비용 | 높음 | 낮음 | B |
| 결론 | — | — | **B 선택** (비용 우선) |
```

**속성-값 표** — 설정/환경 정보:
```
| 항목 | 값 | 비고 |
|------|-----|------|
| 포트 | 8080 | 개발 환경 기준 |
| DB | PostgreSQL 15 | — |
| 캐시 | Redis 7 | 세션용 |
```

**상태 추적 표** — 진행 현황:
```
| 기능 | 담당 | 상태 | 완료일 |
|------|------|------|--------|
| 로그인 | 홍길동 | ✅ 완료 | 03-28 |
| 대시보드 | 김철수 | 🔄 진행 중 | — |
| 리포트 | — | ⬜ 미착수 | — |
```

## 프로젝트 문서 유형별 가이드

### 프로젝트 개요 페이지

```
📌 [콜아웃] 한줄 요약: 무엇을, 왜 만드는가
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 현황
| 항목 | 값 |
|------|-----|
| 상태 | 개발 중 / 운영 중 / 종료 |
| 현재 버전 | v2.3.0 |
| 기술 스택 | TypeScript, React, PostgreSQL |
| 저장소 | (링크) |

## 목적과 배경
(2~3문장으로 핵심만. 장문 배경은 토글로)

## 주요 기능
- 기능 A: 한줄 설명
- 기능 B: 한줄 설명
- 기능 C: 한줄 설명

## 시스템 구성
(다이어그램 또는 간단한 구성도)

▶ [토글] 상세 아키텍처
```

### 기능 명세 페이지

```
## 기능 A
▶ [토글] 상세 명세
  - 목적: 한줄
  - 입력/출력
  - 비즈니스 규칙 (표로 정리)
  - UI 흐름 또는 API 스펙
```

## 안티패턴 — Notion 문서에서 하지 말 것

| 안티패턴 | 결과 | 대안 |
|---------|------|------|
| 텍스트만으로 페이지 가득 채움 | 읽기 싫은 텍스트 벽 → 아무도 안 읽음 | 표, 콜아웃, 토글로 시각적 분리 |
| 모든 정보를 한 페이지에 넣음 | 스크롤이 끝없이 길어짐 | 주제별 하위 페이지 분리 + 링크 |
| 빈 섹션 남겨두기 ("추후 작성") | 문서 신뢰도 하락, 미완성 인상 | 내용 없으면 섹션 자체 제거 |
| 글머리 기호 10개 이상 나열 | 핵심이 묻힘, 시각적 단조로움 | 5개 이상이면 그룹핑 또는 표로 전환 |
| 모든 텍스트에 볼드/이탤릭 | 강조가 의미 없어짐 (전부 강조 = 강조 없음) | 핵심 키워드 1~2개만 볼드 |
| 코드/명령어를 일반 텍스트로 | 복사 불편, 가독성 저하 | 인라인 코드(`) 또는 코드 블록 사용 |
| 날짜를 상대적으로 기록 ("어제", "다음 주") | 시간이 지나면 의미 없어짐 | 절대 날짜 사용 (2026-03-30) |
| 스크린샷만으로 설명 | 검색 불가, 수정 불가, 접근성 문제 | 텍스트로 정보 작성 + 스크린샷은 보조 |
| 같은 정보를 여러 페이지에 중복 | 하나만 업데이트하면 나머지 불일치 | 원본 1곳 + 나머지는 링크 |
| 토글 제목이 "자세히 보기" | 뭐가 들어있는지 모름 → 클릭 안 함 | 내용을 예측할 수 있는 구체적 제목 |

## AI가 Notion 문서 작성 시 체크리스트

### 작성 전 확인
- [ ] 이 정보를 가장 잘 표현하는 블록 타입은? (표? 목록? 토글?)
- [ ] 페이지 상단에 한눈에 파악할 수 있는 요약/현황이 있는가?
- [ ] 계층 구조가 논리적인가? (H2 → H3 순서, 건너뛰기 없음)

### 작성 후 검증
- [ ] 3초 안에 페이지의 전체 구조를 파악할 수 있는가? (스캔 가능성)
- [ ] "있어도 없어도 되는" 문장이 있는가? → 삭제
- [ ] 5줄 이상 텍스트가 연속되는 구간이 있는가? → 표/목록/토글로 변환
- [ ] 10개 이상 나열이 있는가? → 그룹핑 또는 표로 전환
- [ ] 빈 섹션이 있는가? → 제거
- [ ] 날짜가 상대적으로 쓰여있는가? ("내일", "지난주") → 절대 날짜로
- [ ] 코드/명령어가 일반 텍스트로 쓰여있는가? → 코드 블록으로

## 에이전트 연동

- 사용자가 "Notion에 정리해줘"라고 요청하면 이 규칙에 따라 **Notion 마크다운 형식**으로 콘텐츠를 생성
- Notion API 사용이 가능하면 API로 직접 업데이트, 불가하면 **복사해서 붙여넣을 수 있는 형태**로 제공
- 기존 페이지 업데이트 시: 전체 덮어쓰기가 아니라 **변경된 섹션만** 식별하여 업데이트
- `/CLAUDE.md`에 Notion 페이지 ID가 정의되어 있으면 해당 페이지에, 없으면 텍스트로 제공
- 문서 유형(릴리즈 노트, 기능 명세, 회의록 등)을 판단하여 위 **유형별 가이드**의 구조를 자동 적용
