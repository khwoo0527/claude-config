# rules/{tech}.md 작성 가이드

> 이 파일은 기술 스택 규칙 파일을 작성할 때 참조하는 **메타 템플릿**입니다.
> 새로운 rules/{tech}.md를 생성할 때 이 구조와 기준을 따르면, 모든 에이전트가
> 해당 기술의 **시니어 개발자 수준**으로 동작하고, 바이브 코딩 시 **최고 품질의 결과물**을 보장합니다.

---

## 작성 워크플로우 (필수)

> rules 파일은 한 번에 쓰고 끝내는 게 아니다.
> **3단계 작성 → 자가 검증 → 점진적 개선** 사이클을 따른다.

### Step 1: 기술 고유 영역 발굴 (작성 전)

아래 템플릿의 **공통 필수 섹션**은 모든 기술에 적용되는 최소 골격이다.
하지만 모든 기술에는 **그 기술만의 고유 영역**이 있다.

작성 전에 반드시 다음 질문에 답한다:

| 질문 | 목적 |
|------|------|
| 이 기술에서 **초보가 가장 많이 실수하는 영역** 3~5개는? | 안티패턴 + 실전 함정 발굴 |
| 이 기술의 **플랫폼/환경별 차이**가 있는가? | 호환성 규칙 도출 |
| 이 기술에서 **성능 병목이 자주 발생하는 지점**은? | 성능 섹션 구체화 |
| 이 기술의 **보안 취약점 Top 5**는? | 보안 섹션 구체화 |
| 공통 필수 섹션 외에 **이 기술에만 필요한 고유 섹션**은? | 기술 고유 섹션 추가 |

**기술 고유 섹션 예시** (이것은 예시일 뿐, 기술마다 다르다):
- 프론트엔드 UI: 애니메이션, 키보드 처리, 접근성, 반응형, 상태 관리 심화
- 백엔드 API: 인증/인가 패턴, Rate Limiting, 데이터베이스 쿼리 최적화, 마이그레이션
- 데이터베이스: 인덱싱 전략, 접근 제어 정책, 트랜잭션 패턴, 마이그레이션
- 모바일: 앱 생명주기, 푸시 알림, 딥링크, 오프라인 대응

발굴한 기술 고유 섹션을 공통 필수 섹션에 **추가하여** 목차를 완성한 후 작성을 시작한다.

### Step 2: 작성 + 자가 검증

1. Step 1에서 완성한 목차를 기반으로 전체 내용을 작성한다
2. **🔴 프로젝트 종속 검증 (필수 — 건너뛸 수 없음)**
   - 작성 완료 후 **반드시** 파일 전체에서 프로젝트 도메인 용어를 검색한다
   - `grep -i` 등으로 프로젝트 테이블명, 도메인 용어, 프로젝트 고유명사를 검색
   - 1건이라도 발견되면 범용 용어로 교체한 후 다음 단계로 진행
   - **범용 예시 기준**: `teams`, `team_members`, `tasks`, `profiles`, `categories` 등 어떤 프로젝트에서든 통용되는 이름
   - **금지 예시**: 프로젝트 DB 테이블명, 프로젝트 고유 도메인 용어, 프로젝트명
3. 아래 품질 체크리스트로 자가 검증한다
4. **Level 2(예방력) 75점 미만이면 배포 금지** — 보강 후 재검증

> **첫 작성 최소 기준은 Level 2 충족(75점)이다.**
> Level 1만 충족(형식만 갖춤)은 60점이며, 이 상태로 코드 작성을 시작하면 실전에서 버그가 터진다.
> Level 3(검증력)은 실전 경험이 쌓이면서 보강한다. Level 4는 점진적 개선으로 도달한다.
>
> **자가 검증 시 핵심 질문**: "이 규칙만 읽고 코드를 작성했을 때, 흔한 실전 문제가 사전에 방지되는가?"
> 이 질문에 "아니오"면 Level 2 미달이다.

### Step 3: 점진적 개선 (작업하면서 발전)

rules 파일은 **살아있는 문서**다. 작업 중 아래 상황이 발생하면 AI가 **능동적으로** 규칙을 업데이트한다:

| 상황 | 행동 |
|------|------|
| 작업 중 규칙에 없는 함정/버그를 발견 | 해당 섹션에 안티패턴 또는 주의사항 추가 |
| 사용자가 코드 품질을 지적 | 원인을 분석하고 재발 방지 규칙 추가 |
| 같은 실수를 2번 이상 반복 | 규칙이 부족한 것 — 구체적 코드 예시와 함께 보강 |
| 새로운 고급 패턴을 적용 | 해당 패턴을 규칙에 추가 (다음에도 적용하기 위해) |
| 라이브러리/프레임워크 버전 업데이트 | 변경된 권장사항 반영 |

> **핵심 원칙: 사용자가 모든 기술을 알고 있는 것이 아니다.**
> AI가 해당 기술의 전문가로서 문제를 발견하면 사용자에게 알리고,
> 규칙을 개선하여 같은 문제가 반복되지 않게 한다.

## 작성 원칙

### 1. 바이브 코딩 최적화
- 규칙은 **구체적이고 실행 가능**해야 합니다. "좋은 코드를 작성하세요" ❌ → "컴포넌트는 단일 책임 원칙에 따라 하나의 역할만 담당. 200줄 초과 시 분리" ✅
- AI가 코드를 생성할 때 **즉시 적용할 수 있는 수준**으로 작성합니다.
- "이렇게 하지 마세요" (안티패턴)을 반드시 포함합니다 — AI가 흔히 생성하는 나쁜 패턴을 방지합니다.
- **코드 예시를 풍부하게** 포함합니다 — AI는 규칙 설명보다 코드 예시에서 패턴을 더 잘 학습합니다.

### 2. 시니어 수준 전문성
- 공식 문서에 나오는 기본 내용만 쓰지 않습니다. **그건 50점짜리입니다.**
- **실무에서 겪는 함정, 성능 이슈, 보안 취약점**을 포함합니다.
- **고급 패턴과 실전 활용법**을 코드 예시와 함께 제시합니다.
- "왜 이렇게 해야 하는지" 이유를 간결하게 덧붙입니다.
- 해당 기술의 **최신 권장 사항**을 반영합니다.
- **안티패턴은 테이블 형식**으로 정리합니다 (안티패턴 | 이유 | 대안).
- **안티패턴의 "이유"는 구체적인 실패 시나리오**여야 합니다. "비효율적이다" ❌ → "N+1 쿼리로 목록 조회 시 응답 10초 초과" ✅

### 2-1. 실전 검증 필수
- 규칙에 쓴 코드 예시가 **실제 환경에서 동작하는지** 검증한다. 이론만 맞고 실전에서 깨지는 코드 예시는 독이다.
- 예: `storage: AsyncStorage` 예시를 제시했지만 웹에서 세션 유지가 안 된다면, 그 예시는 잘못된 것이다.
- **"이 규칙을 따랐는데도 문제가 발생한" 경우가 곧 규칙의 결함**이다 — 즉시 보완한다.
- 규칙의 검증 방법: "이 규칙만 읽고 코드를 작성했을 때, 지금까지 겪은 모든 버그가 사전에 방지되는가?"

### 3. 꼼꼼하고 상세하게
- 빠뜨리는 영역이 없도록 아래 템플릿의 **모든 필수 섹션**을 채웁니다.
- 각 섹션에 최소 **3~5개의 구체적 규칙 + 코드 예시**를 포함합니다.
- 애매한 표현 대신 **명확한 기준**을 제시합니다.
- 네이밍 컨벤션은 **테이블로 정리**합니다 (종류 | 컨벤션 | 예시).
- 라이브러리 추천도 **테이블로 정리**합니다 (용도 | 권장 | 이유).

### 4. 프로젝트 독립성 (🔴 반복 위반 주의)
- 특정 프로젝트에 종속된 내용을 넣지 않습니다.
- **코드 예시도 포함**: `clubs`, `schedule_events` 같은 프로젝트 테이블명을 코드 예시에 그대로 사용하면 안 됩니다. `teams`, `tasks` 등 범용 이름을 사용합니다.
- **흔한 실수**: 프로젝트에서 겪은 실전 경험을 규칙에 추가할 때, 프로젝트 코드의 테이블명/변수명을 그대로 복사합니다. 경험은 반영하되 용어는 범용화해야 합니다.
- 해당 기술 스택을 사용하는 **모든 프로젝트**에 적용 가능해야 합니다.
- 프로젝트별 설정은 CLAUDE.md에 위임합니다.
- **검증**: 작성/수정 후 반드시 `grep`으로 프로젝트 도메인 용어 0건 확인.

## 템플릿 구조

> **필수 섹션**은 반드시 포함합니다. **선택 섹션**은 해당 기술에 적용되는 경우만 포함합니다.
> 각 섹션의 설명 아래 `[참고: ...]`는 작성 가이드이며, 실제 파일에는 포함하지 않습니다.

````markdown
---
paths:
  - "{해당 기술의 파일 패턴}"
  - "예: **/*.ts, **/*.tsx, **/*.py 등"
---

# {기술명} 개발 규칙

> 이 파일은 {기술명} 프로젝트에 범용적으로 적용되는 시니어 수준 규칙입니다.
> 프로젝트별 설정, 빌드 명령, 폴더 구조는 `/CLAUDE.md`에 정의합니다.

## 핵심 원칙
[필수] 이 기술의 철학과 핵심 설계 원칙 3~5개.
[참고: 단순 나열이 아니라 "왜 이 원칙이 중요한지" 한 줄 설명을 덧붙인다.
       예: "any 금지 — any는 타입 시스템을 무력화한다. unknown을 쓰고 타입 가드로 좁혀라."]

## 프로젝트 구조
[필수] 권장 폴더/파일 구조를 트리 다이어그램으로 제시.
[참고: 기능 기반 vs 레이어 기반 등 구조 철학을 설명하고,
       파일 배치 규칙, 모듈/컴포넌트 분리 기준을 구체적으로 명시한다.]

## 네이밍 컨벤션
[필수] 테이블 형식으로 정리.
[참고: 파일/폴더, 변수/함수, 클래스/타입, 상수, boolean, 이벤트 핸들러 등
       모든 종류를 빠짐없이 포함한다. 의미 기반 접두어/접미어도 테이블로 정리한다.
       예: | boolean | is/has/can/should 접두어 | isLoading, hasPermission |]

## 코드 포맷팅
[필수] 들여쓰기, 따옴표, 세미콜론, 줄 길이, 후행 쉼표, 중괄호, 빈 줄 규칙.
[참고: Prettier/formatter 설정값 기준으로 명시한다.
       "팀마다 다름"이 아니라 구체적 권장값을 제시한다.]

## import/export 규칙
[필수] import 정렬 순서를 코드 예시로 제시.
[참고: 외부 → 내부 절대경로 → 상대경로 순서.
       named vs default export 정책, 타입 전용 import/export, re-export 규칙을 명시한다.]

## 파일 내부 코드 순서
[필수] 파일 내에서 코드가 나열되는 순서.
[참고: import → 타입 → 상수 → 헬퍼 → 메인 export 등.
       이 순서를 따르면 파일을 위에서 아래로 읽기만 해도 구조가 파악된다.]

## 코드 구조 & 패턴
[필수] 권장 패턴을 코드 예시와 함께 제시.
[참고: 기본 패턴만이 아닌 고급/실전 패턴을 반드시 포함한다.
       - 기본: 함수 선언, 클래스/모듈 구조, 상태 관리, 비동기 처리
       - 고급: 해당 기술 특유의 고급 패턴 (예: TypeScript의 Branded Type, Discriminated Union,
              React의 Compound Component, Python의 Descriptor Protocol 등)
       - 실전: 불변 업데이트 패턴, 에러 전파 패턴, 의존성 주입 등]

## 안티패턴 (하지 말 것)
[필수] 테이블 형식: | 안티패턴 | 실패 결과 | 대안 |
[참고: 세 카테고리로 나누어 작성한다:
       1. 해당 기술의 일반적 안티패턴
       2. AI가 흔히 생성하는 실수 (바이브 코딩 특화)
       3. 실전에서 발견된 지뢰 (프로젝트 경험에서 축적)
       **"이유"가 아닌 "실패 결과"를 명시한다:**
       - Bad: "비효율적이다", "좋지 않다"
       - Good: "N+1 쿼리로 응답 10초 초과", "웹에서 세션 유실", "NULL로 필터 누락"
       AI 실수는 실제로 자주 발생하는 것을 구체적으로 나열한다.]

[필수] 해당 기술의 null/undefined/None/nil 처리 베스트 프랙티스.
[참고: 옵셔널 체이닝, 타입 가드, 기본값 패턴 등을 코드 예시와 함께 제시한다.]

## 비동기 처리
[필수] 해당 기술의 비동기 패턴.
[참고: async/await, 병렬 실행, 취소, 타임아웃, 에러 처리를 코드 예시와 함께 제시한다.]

## 에러/예외 처리
[필수] 에러 처리 패턴을 코드 예시와 함께 제시.
[참고: 기본 try-catch뿐 아니라 커스텀 에러 클래스, 에러 전파 전략,
       사용자 에러 vs 내부 에러 구분, Result 패턴 등 실전 패턴을 포함한다.]

## 보안
[필수] 해당 기술에서 주의할 보안 취약점.
[참고: OWASP 관련 항목, 입력 검증, 인증/인가, 민감 정보 관리,
       해당 기술 특유의 보안 위험 (XSS, SQL Injection, Command Injection 등)을 구체적으로 명시한다.]

## 성능 최적화
[필수] 해당 기술의 성능 함정과 최적화 기법.
[참고: 카테고리별로 나누어 작성한다:
       - 렌더링/실행 성능
       - 메모리 관리
       - 번들/로딩 최적화
       - 캐싱 전략
       "측정 후 적용" 원칙도 명시한다.]

## 테스트
[필수] 테스트 전략, 도구, 규칙.
[참고: 테스트 파일 위치, 네이밍, 유닛/통합/E2E 기준, 코드 예시를 포함한다.]

## 스타일링
[선택 — UI 기술인 경우 필수] 스타일링 방법론.
[참고: 반응형 기준, 테마/디자인 시스템, 접근성(a11y) 기준을 포함한다.]

## 실전 함정 & 지뢰
[필수 — Level 2 핵심 섹션] 실제 프로젝트에서 발견된 함정을 구체적 시나리오로 기술.
[참고: 안티패턴과 다른 점 — 안티패턴은 "하지 말 것", 함정은 "정상적으로 보이지만 특정 조건에서 깨지는 것".

       예 (다양한 기술에서):
       | 증상 | 원인 | 해결 | 오답 (하지 말 것) |
       |------|------|------|-------------------|
       | 목록 API 응답 10초 초과 | N+1 쿼리 (항목마다 추가 쿼리 실행) | JOIN 또는 배치 쿼리로 변경 | 타임아웃을 30초로 늘림 |
       | 로그인 후 새로고침 시 로그아웃 | 세션 토큰이 메모리에만 저장 | 쿠키 또는 안전한 스토리지에 저장 | 매 요청마다 재로그인 |

       처음 작성 시 빈 섹션이어도 된다 — 실전 경험이 쌓이면서 채워진다.
       **이 섹션이 풍부할수록 Level 4(시니어 수준)에 가까워진다.**]
````

## 품질 체크리스트

> **핵심 원칙: 규칙의 품질은 "섹션이 채워졌는가"가 아니라 "이 규칙을 따랐을 때 실전 문제를 방지할 수 있는가"로 측정한다.**
>
> 형식만 갖춘 문서는 50점이다. 공식 문서를 번역한 수준은 60점이다.
> 실전에서 겪은 함정이 반영되고, 규칙 준수를 검증할 수 있어야 80점 이상이다.

### Level 1: 형식 (40~60점 — 이 수준은 부족하다)

> "섹션이 채워져 있고 코드 예시가 있는가?" — 형식만 갖추면 이 수준.
> 공식 문서 내용을 정리한 수준. **실전에서 버그를 방지하지 못한다.**

- [ ] 모든 공통 필수 섹션이 채워졌는가?
- [ ] 각 섹션에 최소 3개 이상의 구체적 규칙이 있는가?
- [ ] 코드 예시가 주요 섹션에 포함되었는가? (최소 10개 코드 블록)
- [ ] 기술 고유 섹션이 발굴되어 추가되었는가?
- [ ] 네이밍 컨벤션/라이브러리 추천이 테이블로 정리되었는가?
- [ ] **🔴 프로젝트 종속 검증 완료**: 파일 전체에서 프로젝트 도메인 용어를 검색(grep)하여 0건 확인했는가? (코드 예시의 테이블명, 변수명, 도메인 용어 포함)

### Level 2: 예방력 (60~80점 — 첫 작성 최소 기준)

> "이 규칙을 따르면 흔한 실전 문제를 사전에 방지할 수 있는가?"
> **핵심: 안티패턴이 이론이 아니라 실제 발생하는 구체적 시나리오여야 한다.**

- [ ] **실전 함정/지뢰**가 구체적 시나리오와 함께 기술되었는가?
  - Bad: "인증을 잘 설정하세요"
  - Good: "JWT 토큰 만료 시 자동 갱신이 없으면 사용자가 30분마다 재로그인 → 리프레시 토큰 흐름 구현 필수"
- [ ] **플랫폼/환경별 차이**가 구체적 코드 분기와 함께 있는가?
  - Bad: "환경마다 차이가 있을 수 있다"
  - Good: "Windows에서 경로 구분자가 `\`이므로 `path.join()` 필수 → 문자열 연결로 경로 만들면 Linux에서 깨짐"
- [ ] **안티패턴 테이블**의 각 항목이 "실제로 이렇게 하면 뭐가 깨지는지" 결과를 명시하는가?
  - Bad: "SELECT * 는 비효율적이다"
  - Good: "SELECT * + 대용량 BLOB 컬럼 = 목록 조회 시 응답 5초 초과 → 필요한 컬럼만 명시"
- [ ] **"왜"가 명시되었는가?** — 규칙마다 "이렇게 안 하면 뭐가 깨지는지" 한 줄이 있는가?
- [ ] **기본값/초기값 함정**이 포함되었는가? (DB 컬럼 기본값, 라이브러리 옵션 기본값 등)
- [ ] **고급 패턴**이 "언제 쓰고 언제 쓰지 않는지" 판단 기준과 함께 있는가?

### Level 3: 검증력 (80~90점 — 실전 배포 가능 수준)

> "규칙을 제대로 따랐는지 확인할 수 있는 구체적 검증 절차가 있는가?"
> **핵심: 체크리스트, 검증 명령, 테스트 시나리오가 있어야 한다.**

- [ ] **검증 체크리스트**가 있는가? (새 테이블 생성 시, 새 화면 추가 시, 배포 전 등 상황별)
  - 예: "새 API 엔드포인트 체크리스트: 인증 적용 ✓ → 입력 검증 ✓ → 에러 응답 형식 ✓ → 다른 역할로 접근 테스트 ✓"
- [ ] **검증 명령/코드**가 제시되었는가? (복붙해서 바로 실행 가능한 수준)
  - 예: "권한 검증: `curl -H 'Authorization: Bearer {user_token}' /api/admin/users` → 403 반환되는지 확인"
- [ ] **실패 모드**가 문서화되었는가? ("이 규칙을 안 지키면 구체적으로 어떤 에러/장애가 발생하는지")
  - 예: "status 기본값 미설정 → INSERT 시 NULL → `WHERE status = 'active'` 필터에 안 걸림 → 데이터가 조회에서 누락"
- [ ] **우회/핵 금지 원칙**이 명시되었는가?
  - "보안 메커니즘을 우회하지 않는다 — 인증/인가 자체를 올바르게 설정하는 것이 정석"
  - "증상 해결(땜빵)이 아닌 근본 원인을 먼저 진단한다"
- [ ] **에러 발생 → 디버깅 → 수정 흐름**이 정의되었는가?
  - 예: "API 500 에러 → 1) 서버 로그에서 스택 트레이스 확인 → 2) 쿼리/로직 오류 특정 → 3) 입력 데이터 검증 → 4) 로직 수정 또는 쿼리 최적화"

### Level 4: 실전 검증 완료 (90~100점 — 시니어 수준)

> "실제 프로젝트에서 이 규칙을 적용하고, 발견된 문제가 피드백 루프로 반영되었는가?"
> **핵심: 규칙이 실전 경험으로 검증되고 계속 진화하고 있는 증거가 있어야 한다.**

- [ ] **실전에서 발견된 함정**이 날짜와 함께 기록되어 있는가? (개선 이력)
  - 예: `<!-- 2026-03-27: JWT 만료 후 API 호출 시 무한 로딩 발견 → 토큰 갱신 흐름 규칙 추가 -->`
- [ ] **이 규칙만 읽고 코드를 작성했을 때, 오늘 겪은 문제가 발생하지 않는가?** (최종 리트머스 테스트)
- [ ] **규칙 간 교차 참조**가 있는가? (예: DB 규칙의 접근 제어 함정이 API 규칙의 인증 흐름과 연결)
- [ ] **빌드/배포 전 검증 절차**가 있는가? (Sprint 완료 = 빌드 통과가 아니라 실환경 테스트 통과)
- [ ] AI가 이 규칙만 읽고도 **시니어 개발자가 작성한 것처럼 보이는 코드를 생성할 수 있는가?**

### 점수 매기는 방법

> **기존 문제점**: 섹션 존재 여부만 체크하여 80~90점을 줬지만, 실전에서 버그를 방지하지 못함.
> **개선**: 각 Level의 체크리스트를 **"실제로 문제를 방지할 수 있는가?"** 관점에서 엄격하게 평가.

1. Level 1 항목을 모두 충족 → 60점 (형식만 갖춤, 부족한 수준)
2. Level 2 항목을 모두 충족 → 75점 (예방력 있음, 실전 배포엔 부족)
3. Level 3 항목을 모두 충족 → 85점 (검증 가능, 실전 배포 가능)
4. Level 4 항목을 모두 충족 → 95점+ (실전 검증 완료, 시니어 수준)

**각 Level 내에서 부분 충족 시 비례 감점.** 예: Level 2를 절반만 충족하면 67점.
**상위 Level 항목이 있어도 하위 Level이 미충족이면 하위 점수에 제한됨.**
예: Level 3 항목이 있어도 Level 1이 미충족이면 60점을 넘을 수 없음.

## 작성 순서 권장

여러 기술 스택의 rules를 생성할 때:

1. **언어 규칙 먼저** (예: `typescript.md`, `python.md`)
2. **프레임워크 규칙** (예: `react-native.md`, `fastapi.md`)
3. **인프라/서비스 규칙** (예: `supabase.md`, `firebase.md`)

언어 규칙이 프레임워크의 기반이 되므로, 이 순서를 따르면 일관성이 유지됩니다.
프레임워크 규칙에서는 언어 규칙과 중복되는 내용을 반복하지 않고 "typescript.md 참조"로 위임합니다.

## 레퍼런스 품질 기준

완성된 rules 파일은 아래와 동등한 수준이어야 합니다:
- Google Style Guide (해당 언어 버전) — 형식/컨벤션 참고
- Airbnb JavaScript/React Style Guide — 규칙의 구체성 참고
- 해당 기술 공식 문서의 Best Practices 섹션 — 기본 내용 커버 확인
- **시니어 개발자가 코드 리뷰에서 지적하는 수준의 규칙** — 핵심 기준
- **프로덕션 장애 포스트모템에서 나오는 교훈** — 실전 함정의 출처

**"공식 문서를 번역한 수준"은 60점입니다. "실제 프로젝트에서 겪은 함정과 해결책"이 포함되어야 80점 이상입니다.**
**"이 규칙을 따랐는데도 버그가 났다"면 규칙이 부족한 것이지, 따르는 사람의 문제가 아닙니다.**

## 점진적 개선 프로세스

> rules 파일은 한 번 쓰고 끝나는 문서가 아니다.
> 작업하면서 발견한 함정, 사용자 피드백, 새로운 패턴이 축적되어 전문가 수준으로 성장한다.

### 개선 트리거 (이 상황이 발생하면 규칙을 업데이트한다)

```
작업 중 버그/함정 발견
  → "실전 함정 & 지뢰" 섹션에 증상/원인/해결/오답 추가

규칙을 따랐는데도 버그 발생
  → 규칙의 결함이다 — 해당 규칙을 즉시 보완 + 검증 체크리스트에 항목 추가
  → 이것이 가장 중요한 트리거다. 규칙이 실전을 커버하지 못하는 증거.

사용자가 코드 품질 지적
  → 원인 분석 → 재발 방지 규칙 + Good/Bad 코드 예시 추가

우회/땜빵으로 해결한 후 정석으로 재수정한 경우
  → "우회/핵 금지 원칙" 섹션에 해당 사례 추가 (왜 우회가 잘못이었는지)

같은 실수 2회 이상 반복
  → 규칙이 부족하거나 모호한 것 → 구체적 코드 예시로 보강

새로운 고급 패턴 적용 성공
  → 패턴 섹션에 추가 (다음에도 같은 수준으로 코드 생성하기 위해)

라이브러리/프레임워크 버전 변경
  → 변경된 권장사항, deprecated API 반영
```

### AI의 역할

- **사용자가 모든 기술 스택을 깊이 아는 것은 아니다**
- AI는 해당 기술의 전문가로서, 문제를 발견하면 사용자에게 알리고 규칙 개선을 제안한다
- 사용자가 "이거 왜 이래?" 라고 물을 때까지 기다리지 않는다 — **능동적으로** 개선한다
- 규칙 업데이트 시 변경 사유를 간단히 남긴다 (왜 이 규칙이 추가되었는지 이후 문맥 이해를 위해)
