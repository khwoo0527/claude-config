# PRD (요구사항 정의서) 작성 가이드

> 이 파일은 `docs/prd.md`를 작성하거나 검토할 때 참조하는 품질 기준입니다.
> PRD는 prd-to-roadmap 에이전트의 입력이 되므로, 이 기준을 충족해야 고품질 ROADMAP과 Sprint가 나옵니다.

---

## PRD의 역할

```
PRD (요구사항)  →  ROADMAP (Phase/Sprint 구조)  →  Sprint 명세  →  구현
     ↑ 이 문서가 모든 것의 시작점
```

- **CLAUDE.md**: 기술 스택, 빌드 명령, 프로젝트 구조 (How)
- **PRD**: 무엇을 만들고, 왜 만들고, 누가 쓰고, 어떻게 동작하는지 (What & Why)
- **ROADMAP**: 어떤 순서로 만드는지 (When)

---

## 필수 섹션

### 1. 제품 개요
- 한 줄 요약
- 풀고자 하는 문제 (Pain Point)
- 타겟 사용자 (페르소나별 역할, 니즈)
- 성장 전략 / 비즈니스 모델

### 2. 사용자 플로우
> **PRD에서 가장 중요한 섹션.** 기능 나열보다 사용자의 여정이 먼저.

각 핵심 시나리오를 **사용자 관점의 Step-by-Step 흐름**으로 작성:

```
예시: 동호회 가입 플로우
1. 비회원이 앱에 접속한다
2. 동호회 목록을 탐색/검색한다
3. 관심 있는 동호회를 탭하여 소개를 본다
4. "가입 신청" 버튼을 탭한다
5. 로그인 화면으로 이동 (미로그인 시)
6. 가입 인사말을 작성하고 신청한다
7. "대기 중" 상태로 알림을 기다린다
8. 클럽장/운영진이 승인하면 알림을 받는다
9. 동호회 내부 (일정, 멤버 등)에 접근 가능해진다
```

작성할 플로우 목록:
- 최초 가입/로그인 플로우
- 동호회 탐색 → 가입 플로우
- 동호회 생성 플로우 (클럽장)
- 일정 생성 플로우 (운영진)
- 일정 참석/대기 플로우 (멤버)
- 알림 수신 → 행동 플로우
- 관리 플로우 (멤버 승인, 강퇴, 역할 변경)

### 3. 기능 상세
각 기능을 아래 구조로 작성:
- **기능 설명**: 무엇을 하는 기능인지
- **사용자 유형**: 누가 사용하는지 (역할별 권한)
- **비즈니스 규칙**: 상세 로직 (자동 마감, 대기 승격, 패널티 등)
- **엣지 케이스**: 예외 상황과 처리 방법
- **MVP 우선순위**: Must-have / Should-have / Nice-to-have

### 4. 데이터 모델
> CLAUDE.md에는 테이블명/관계만, PRD에 필드 상세를 작성한다.

각 테이블:
- 테이블명, 설명
- 필드 목록 (이름, 타입, 필수 여부, 설명)
- 관계 (FK, 1:N, N:M)
- 인덱스 전략
- 확장 고려사항

### 5. 알림/이벤트 설계
각 알림:
- 트리거 (무엇이 발생하면)
- 수신 대상
- 내용 (제목, 본문)
- 행동 (탭하면 어디로)
- 우선순위 (즉시 / 배치)

### 6. 우선순위 분류 (MoSCoW)
- **Must-have**: 이거 없으면 MVP가 아님
- **Should-have**: 있으면 좋지만 없어도 출시 가능
- **Nice-to-have**: 여유 있으면 추가
- **Won't-have (이번에)**: 명시적으로 안 하는 것

### 7. 엣지 케이스 & 예외 처리
시스템에서 발생할 수 있는 예외 상황들:
- 사용자 탈퇴/삭제 시 데이터 처리
- 클럽장 탈퇴 시 동호회 처리
- 동호회 삭제/비활성화 시 멤버 처리
- 네트워크 오프라인 시 동작
- 동시 참석 신청 (race condition)
- 정원 초과 시 대기 → 참석 승격 상세 로직

### 8. 장기 비전 (Phase별)
- Phase 1 (MVP): 핵심 기능만
- Phase 2~N: 확장 기능 (결제, 다종목, 커뮤니티 등)
- 각 Phase의 핵심 가치와 의존성

### 9. 비기능적 요구사항
- 성능 (로딩, 응답, FPS)
- 확장성 (동시 접속, 데이터 규모)
- 보안 (인증, 접근 제어, 암호화)
- 접근성 (a11y)
- 플랫폼 지원 (웹, Android, iOS)

### 10. 경쟁 분석 (선택)
- 유사 서비스와 차별점
- 벤치마킹할 UX

### 11. 성공 지표 (KPI)
- 정량적 측정 기준 (DAU, 동호회 수, 참석률 등)
- MVP 성공 판단 기준

## 품질 체크리스트

### 기본 (70점)
- [ ] 모든 필수 섹션이 채워졌는가?
- [ ] 타겟 사용자가 역할별로 정의되었는가?
- [ ] MVP 범위가 명확한가?

### 시니어 (85점)
- [ ] **사용자 플로우**가 핵심 시나리오별로 작성되었는가? (기능 나열만 X)
- [ ] **비즈니스 규칙**이 구체적으로 명시되었는가? (대기 승격, 자동 마감 등)
- [ ] **엣지 케이스**가 5개 이상 식별되었는가?
- [ ] **MoSCoW 우선순위**가 분류되었는가?
- [ ] **데이터 모델** 필드 상세가 포함되었는가?

### 최고 (90점+)
- [ ] **성공 지표(KPI)**가 정량적으로 정의되었는가?
- [ ] **알림 설계**에 트리거/대상/내용/행동이 모두 명시되었는가?
- [ ] **경쟁 분석** 또는 차별점이 포함되었는가?
- [ ] 각 기능에 **역할별 권한**이 명시되었는가?
- [ ] Phase별 장기 비전에 **핵심 가치와 의존성**이 있는가?
- [ ] prd-to-roadmap 에이전트가 이 PRD만 읽고 ROADMAP을 만들 수 있는가?
