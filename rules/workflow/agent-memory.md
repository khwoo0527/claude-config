# Agent Memory 정책 (단일 진실 원천)

> `.claude/agent-memory/{name}/MEMORY.md` 의 **용도·기록 시점·기록 방식**을 정의한다.
> 모든 에이전트는 이 정책을 따라야 하며, 본문 절차는 이 가이드를 참조하기만 한다 (중복 명시 금지).

---

## 1. 핵심 위치 분기

작업 중 무언가 발견했을 때 **이 정보가 다른 프로젝트에서도 통하는가?** 자문하여 위치 결정한다.

| 위치 | 성격 | 누적/리셋 | 담는 것 |
|---|---|---|---|
| **`rules/tech/{tech}.md`** | 범용 기술 노하우 | 누적 (영구) | 다른 프로젝트에서도 통하는 안티패턴/함정/베스트 프랙티스 |
| **`memory/feedback_*.md`** | 범용 사용자 작업 원칙 | 누적 (영구) | 작업 스타일/원칙 (반복 방지) |
| **`agent-memory/{name}/`** | 이 프로젝트 작업 컨텍스트 | **새 프로젝트 시 리셋** | 이 프로젝트만 의미 있는 코드베이스 분석/작업 컨텍스트 |
| **`ROADMAP.md` / `sprint{N}.md` / `deploy.md`** | 프로젝트 공식 진행 일지 | 새 프로젝트 시 새로 작성 | 진행 상태 (진실 원천) |

> **원칙**: 노하우는 누적 위치(rules/tech)에 가야 가치 있다. agent-memory 는 리셋되므로 거기에 누적해봐야 다음 프로젝트에서 무의미하다.

---

## 2. agent-memory 의 정확한 용도

agent-memory 는 **이 프로젝트만 의미 있는 작업 컨텍스트의 캐시**.

### ✅ 담을 것
- 이 프로젝트 코드베이스 분석 결과 (예: "이 프로젝트는 Supabase Cloud 사용 중, 마이그레이션은 push 방식")
- 이 프로젝트의 특정 모듈/스키마 구조 메모 (다음 호출 때 빠른 컨텍스트 복원용)
- 작업 중인 분석/검토의 중간 산출물 (예: 전문가 리뷰 결과 요약)
- 미해결 리스크/TODO (이 프로젝트 한정)

### ❌ 담지 않을 것
- **다른 프로젝트에서도 통하는 일반 노하우** → `rules/tech/{tech}.md` 로
  - 예: "Supabase 의 `update_updated_at()` 함수 중복 정의 금지" (모든 Supabase 프로젝트 공통)
- **사용자 작업 스타일** → `memory/feedback_*.md` 로
- **공식 진행 상태** (Sprint 번호, Phase 분할, 배포 이력) → ROADMAP/sprint{N}.md/deploy.md 가 진실. agent-memory 에 캐시하면 두 곳 동기화 부담 발생.

> **이유**: 진실 원천이 이미 다른 곳에 있으면 agent-memory 에 캐시하지 않는다. 새 세션에서 그때그때 Read.

## 3. 분기 판단 — 발견 시 어디에 기록?

작업 중 새로 발견된 정보는 다음 순서로 판단:

1. **사용자 작업 스타일/원칙인가?** → `memory/feedback_*.md`
2. **다른 프로젝트에서도 통하는 기술 노하우인가?** → `rules/tech/{tech}.md` 보강
3. **이 프로젝트의 공식 진행 상태/결정 사항인가?** → `CLAUDE.md` / `ROADMAP.md` / `sprint{N}.md` / `deploy.md`
4. **위 어디에도 안 맞는 이 프로젝트 작업 컨텍스트인가?** → `agent-memory/{에이전트명}/MEMORY.md`

### 판단 어려울 때
사용자에게 컨펌:
```
💡 발견: {정보 요약}
이게 다른 프로젝트에서도 통하는 기술 노하우면 rules/tech/{tech}.md 보강,
이 프로젝트만의 컨텍스트면 agent-memory 에 둘게요. 어떤가요?
```

## 4. 기록 시점 — 사용자 컨펌

### 일반 원칙: 컨펌 후 기록
- 작업 도중 가치 있는 정보 발견 시 위 분기 판단 + 사용자에게 컨펌:
  ```
  💡 발견: {정보 요약}
  → {제안 위치} 에 기록할까요? (Y/N)
  ```
- 사용자 Y → 기록. N → 무시.

### 예외: 위임 신호 받은 경우
사용자가 명시적 위임 신호 ("권한 다 OK"/"묻지 말고"/"자리 비울게" 등) 를 준 경우, 그 작업 동안은 컨펌 생략하고 자동 기록한다. 작업 종료 시 사후 보고 필수.

## 5. agent-memory 기록 방식

### 파일 구조
- 위치: `.claude/agent-memory/{에이전트명}/MEMORY.md`
- 형식: 작업 컨텍스트 섹션별 정리 (학습/패턴 형식이 아님)

### 항목 예시
```markdown
## 코드베이스 컨텍스트
- 인프라: {예: Supabase Cloud, Vercel}
- 핵심 모듈/스키마: {예: profiles, clubs, ...}
- 마이그레이션 방식: {예: npx supabase db push}

## 작업 중 분석 결과
- {YYYY-MM-DD} {분석 항목}: {결과 요약}

## 미해결 리스크/TODO
- {항목}: {상태/맥락}
```

### 갱신 vs 추가
- 동일 컨텍스트 갱신 → 기존 라인 수정 (캐시이므로 최신만 유지)
- 새 컨텍스트 → 추가

## 6. 폴더 존재 = 메모리 사용 (단일 신호)

- `.claude/agent-memory/{에이전트명}/` 폴더가 있으면 그 에이전트는 메모리 사용.
- 폴더가 없으면 사용 안 함.
- frontmatter 의 `memory:` 같은 별도 표시는 사용하지 않는다 (이중 진실 방지).

> 새 에이전트가 메모리 필요하면: `agent-memory/{name}/` 폴더 + 빈 `MEMORY.md` 생성. 그 외 메타데이터 불필요.

---

## 7. 새 프로젝트 시작 시 (project-init 정책)

- agent-memory 는 프로젝트 종속이므로 새 프로젝트에서 그대로 가져가지 않는다.
- `project-init` 에이전트가 정리 절차에서 각 폴더의 `MEMORY.md` 를 초기 상태로 리셋.
- **rules/tech/, memory/ 는 그대로 유지** (범용 — 누적 가치).

---

## 8. 로딩 시점

- agent-memory 는 **자동 로딩 X**. 해당 에이전트가 호출될 때 본인이 자기 폴더의 MEMORY.md 를 Read.
- `/InitLoad` 의 메모리 일괄 로딩과는 무관 (`.claude/memory/` 만 InitLoad 대상).

---

## 9. 범용 노하우 누적 절차 (rules/tech/{tech}.md)

agent 가 작업 중 다른 프로젝트에도 통하는 기술 노하우를 발견하면 `rules/tech/{tech}.md` 에 직접 추가한다. 자세한 절차/형식은 [`tech-knowledge.md`](tech-knowledge.md) 참조.

---

## 10. 위반 사례 감지

다음이 보이면 정책 위반 → 사용자에게 보고:
- agent-memory 에 다른 프로젝트에도 통할 일반 노하우가 있음 → `rules/tech/{tech}.md` 로 옮길 것을 제안
- agent-memory 에 ROADMAP/sprint 의 공식 진행 상태가 캐시되어 있음 → 이중 기록 (제거 또는 ROADMAP 만 진실 원천으로)
- 사용자 컨펌 없이 자동 기록된 항목 (위임 신호 외) → 정책 위반
- frontmatter 에 `memory:` 표시가 있는 에이전트 → 폐지된 메타데이터, 제거 권고
