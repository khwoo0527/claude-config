# 세션 시작 시 컨텍스트 로딩 규칙

> 이 규칙은 모든 작업(Sprint, ROADMAP, Phase, 코드 리뷰, 문서 작업 등)에 공통 적용된다.
> 어떤 작업이든 컨텍스트 없이 시작하면 품질이 떨어지고 같은 실수를 반복한다.

> **책임 분리 원칙**:
> - **메모리 로딩 정책** → [`.claude/memory/MEMORY.md`](../../memory/MEMORY.md)
> - **에이전트 메모리 정책** → [`agent-memory.md`](agent-memory.md)
> - **새 프로젝트 정리 책임** → [`.claude/agents/project-init.md`](../../agents/project-init.md)
> - **이 문서**는 "**메모리 외 컨텍스트**(rules, ROADMAP, 작업 유형별 가이드)의 로딩 절차" + "**에이전트 진입 절차**(모든 에이전트 공통)" 를 다룬다.

## 자동 로딩 (시스템이 처리)
- `/CLAUDE.md` — 프로젝트 기술 스택, 구조, 빌드 명령

## 수동 로딩 (에이전트가 직접 읽어야 함)

아래 파일이 로딩되지 않은 상태에서 작업을 시작하지 않는다.

| 순서 | 대상 | 목적 |
|------|------|------|
| 1 | 메모리 일괄 (`/InitLoad` 커맨드) | 사용자 프로필, 작업 스타일, 피드백 — 로딩 방식은 [MEMORY.md](../../memory/MEMORY.md) 가 정의 |
| 2 | CLAUDE.md 의 기술 스택에 해당하는 `rules/tech/{tech}.md` | 기술 스택 전문가로 동작 |
| 3 | 작업 유형에 맞는 `rules/workflow/*.md` | 워크플로우 규칙 확인 |
| 4 | `ROADMAP.md` (있는 경우) | 현재 진행 상황 파악 |

### 작업 유형별 로딩 대상

모든 작업에 전부 읽을 필요는 없다. **작업 유형에 따라 필요한 것만** 읽는다:

| 작업 유형 | 필수 로딩 (메모리 외) | 추가 로딩 (해당 시) |
|----------|---------------------|-------------------|
| **코드 구현** (Sprint 등) | `rules/tech/{tech}.md` → `sprint-workflow.md` | `ROADMAP.md`, `sprint{N}.md` |
| **코드 리뷰** | `rules/tech/{tech}.md` | `sprint-workflow.md` (품질 기준) |
| **문서 작성** (PRD, 정리 등) | `prd-guide.md` 또는 `notion.md` | `ROADMAP.md` |
| **Sprint 계획** | `sprint-workflow.md` → `ROADMAP.md` | `rules/tech/{tech}.md` (Task 설계 시) |
| **단순 질문/탐색** | (메모리만으로 충분) | 질문 관련 rules (해당 시) |
| **새 프로젝트 시작** | `project-init` agent 호출 | (agent 가 정리/세팅 절차 수행 — 별도 사전 로딩 불필요) |
| **rules 작성/수정** (`rules/tech/*` 또는 `rules/workflow/*`) | `rules-guide.md` + `tech-knowledge.md` | 수정 대상이 기존 파일이면 해당 파일 |
| **에이전트 작성/수정** (`agents/*`) | `agent-guide.md` + `agent-memory.md` | 수정 대상이 기존 에이전트면 해당 파일 |
| **.claude 폴더 구조 변경** (새 문서/폴더 추가, 이동, 정리) | `folder-structure.md` | 작업 종류에 따라 `rules-guide.md` / `agent-guide.md` 등 |

> **메모리 로딩은 항상 1번 단계로 선행**. 자세한 메모리 로딩 가이드는 [MEMORY.md](../../memory/MEMORY.md) 참조.

## 로딩 확인 원칙

- 세션 시작 후 첫 작업 요청 시, 위 파일을 아직 읽지 않았으면 **먼저 읽고 나서 작업을 시작**한다
- 사용자가 "규칙 로딩해", ".claude 읽고 와" 등을 요청하면 즉시 전체 로딩을 수행한다
- `/InitLoad` 슬래시 커맨드 사용 시 — 메모리 일괄 자동 로딩 (MEMORY.md 정책 따름) + 본 문서 로딩

## 로딩하지 않았을 때 발생하는 문제

| 로딩 누락 | 실제 발생하는 문제 |
|----------|-----------------|
| `rules/tech/{tech}.md` 미로딩 | 해당 기술의 안티패턴을 모르고 그대로 코드 생성 (예: 하드코딩, 잘못된 네이밍) |
| `sprint-workflow.md` 미로딩 | 품질 기준/보안 원칙 무시, 우회/땜빵 해결책 사용 |
| `ROADMAP.md` 미로딩 | 프로젝트 진행 상황을 모르고 엉뚱한 작업 시작, 이미 완료된 작업 재시도 |
| `rules-guide.md` 미로딩 (rules 작성 시) | 4단계 품질 체크리스트 무시 → Level 1 미만 규칙 양산, 시니어 수준 미달 |
| `agent-guide.md` 미로딩 (에이전트 작성 시) | 5개 에이전트 통일 패턴 위반 (frontmatter 폐지 키 사용, 진입 절차 중복 명시 등) |
| `folder-structure.md` 미로딩 (.claude 구조 변경 시) | 폴더 정체성 위반 (가이드를 templates/ 에, 빈 골격을 rules/ 에 등) |

> 메모리 미로딩 시 발생 문제는 [MEMORY.md](../../memory/MEMORY.md) 참조.

## 에이전트 진입 절차 (모든 에이전트 공통)

> 에이전트(서브 에이전트 포함)는 메인 Claude 의 컨텍스트를 공유하지 않는다 — **자기 인스턴스에서 컨텍스트를 다시 로딩**해야 한다.
>
> 이 절차는 작업 디렉토리의 `CLAUDE.md` 가 자동 로딩되는 메커니즘을 통해 모든 에이전트에 강제된다 (CLAUDE.md 안 "에이전트 진입 절차" 섹션이 본 절차로 위임).

### 진입 절차 (작업 시작 전 필수)

1. **메모리 로딩** — `.claude/memory/MEMORY.md` 의 로딩 방식 따라 메모리 일괄 Read.
2. **기술 룰 로딩** — `CLAUDE.md` 의 기술 스택 보고 해당 `.claude/rules/tech/{tech}.md` Read.
3. **작업 유형별 추가 로딩** — 위 "작업 유형별 로딩 대상" 표 참조하여 필요한 것만.
4. (해당 시) **에이전트 자기 메모리 Read** — `.claude/agent-memory/{에이전트명}/MEMORY.md` (있는 경우).
5. 위 절차가 완료된 후에야 본 에이전트의 작업 절차로 진입.

### 강제 메커니즘

- 모든 프로젝트의 `CLAUDE.md` (또는 [`templates/CLAUDE-TEMPLATE.md`](../../templates/CLAUDE-TEMPLATE.md) 기반 신규 생성분) 에는 "에이전트 진입 절차" 섹션이 포함된다.
- `CLAUDE.md` 는 시스템이 자동 로딩하므로, 어떤 에이전트가 호출되든 본 절차 명령을 받음.
- 새 에이전트가 추가되어도 본문에 별도 명시 불필요 — `CLAUDE.md` 가 모두에 일괄 적용.

### 위반 감지

다음이 보이면 진입 절차 누락 → 사용자에게 보고:
- 사용자 작업 스타일을 모르는 에이전트 (한 번에 여러 파일 수정 등 피드백 무시)
- 기술 룰을 안 따르는 에이전트 (네이밍 컨벤션 위반 등)
- ROADMAP 상태를 모르고 엉뚱한 작업 시작

### Good vs Bad 시나리오

```
❌ Bad — 컨텍스트 없이 바로 코드 작성
사용자: "이 함수 리팩토링해줘"
AI: (rules 안 읽고) 바로 코드 수정 → 네이밍 컨벤션 위반, 공용화 원칙 무시, 파일 3개 동시 수정
사용자: "왜 컨벤션 안 지켜?" / "한 파일씩 하라고 했잖아"
```

```
✅ Good — 컨텍스트 로딩 후 작업
사용자: "이 함수 리팩토링해줘"
AI: MEMORY.md 가이드 따라 메모리 로딩 → rules/tech/{tech}.md (네이밍, 구조) 읽기
    → 파일 1개씩, 컨벤션 준수하여 수정
사용자: "좋아, 다음 파일"
```
