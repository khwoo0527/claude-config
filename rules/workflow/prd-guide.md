# PRD (요구사항 정의서) 작성 가이드

> 이 파일은 `docs/prd.md`를 작성하거나 검토할 때 참조하는 품질 기준입니다.
> PRD는 prd-to-roadmap 에이전트의 입력이 되므로, 이 기준을 충족해야 고품질 ROADMAP과 Sprint가 나옵니다.
> 어떤 기술 스택/플랫폼/도메인이든 적용 가능한 범용 가이드입니다.

---

## PRD의 역할

```
PRD (요구사항)  →  ROADMAP (Phase/Sprint 구조)  →  Sprint 명세  →  구현
     ↑ 이 문서가 모든 것의 시작점
```

| 문서 | 담당 |
|------|------|
| **CLAUDE.md** | 기술 스택, 빌드 명령, 프로젝트 구조 (How) |
| **PRD** | 무엇을 만들고, 왜 만들고, 누가 쓰고, 어떻게 동작하는지 (What & Why) |
| **ROADMAP** | 어떤 순서로 만드는지 (When) |

**PRD가 부실하면 모든 하위 문서가 부실해진다.** ROADMAP이 엉성하고, Sprint 명세가 애매하고, 구현 중 "이거 어떻게 해야 해요?" 질문이 반복된다.

## 핵심 작성 원칙

### 1. 사용자/사용 시나리오 먼저, 기능 나열은 나중에

```
❌ 나쁜 순서: 기능 목록 → (시나리오 대충) → 데이터 모델
✅ 좋은 순서: 사용자 정의 → 사용 시나리오 → 기능 상세 → 데이터 모델
```

> "사용자"는 사람일 수도 있고, API 소비자(다른 시스템)일 수도 있고, CLI 사용자일 수도 있다.
> 프로젝트 유형에 맞게 "누가 이 제품을 쓰는가"를 먼저 정의한다.

### 2. 구체적 > 추상적
모든 규칙/조건에 **숫자, 기준, 조건**을 넣는다.

```
❌ "대량 데이터를 잘 처리해야 한다"
✅ "입력 파일 최대 500MB, 처리 시간 파일당 30초 이내, 메모리 사용 1GB 이하"

❌ "권한이 있는 사용자만 접근 가능"
✅ "role='admin'인 사용자만 삭제 API 호출 가능. 권한 없으면 HTTP 403 반환"
```

### 3. "안 하는 것"을 명시적으로 적는다
Won't-have를 적지 않으면, 구현 중에 "이것도 해야 하나?" 혼란이 생긴다.

```
✅ Won't-have (이번에 안 하는 것):
- 다국어 지원 → Phase 3에서 검토
- 오프라인 모드 → 네트워크 필수로 운영
- 관리자 대시보드 → 별도 프로젝트로 분리
```

### 4. 에이전트가 읽을 수 있는 구조
prd-to-roadmap 에이전트가 PRD를 파싱하여 ROADMAP을 생성한다:
- 마크다운 헤딩으로 섹션을 명확히 구분
- 기능별 우선순위(MoSCoW)를 반드시 태그
- 기능 간 의존성이 있으면 명시 ("이 기능은 인증 기능이 먼저 구현되어야 함")

## 필수 섹션

> 프로젝트 유형에 따라 일부 섹션이 해당되지 않을 수 있다.
> 해당되지 않는 섹션은 "해당 없음"으로 표기하되, **건너뛰지 않고 판단 근거를 남긴다.**

### 1. 제품 개요
- 한 줄 요약
- 풀고자 하는 문제 (Pain Point)
- 타겟 사용자/소비자 (역할, 니즈)
- 제품 유형 (웹 앱 / 모바일 앱 / CLI / 라이브러리 / API / 데스크톱 앱 등)

**Good vs Bad:**

```
❌ Bad:
"관리 시스템입니다." / "편리한 도구입니다."

✅ Good (데스크톱 도구):
"레거시 DB → 신규 DB 마이그레이션을 자동화하는 데스크톱 도구.
Pain Point: 수작업 변환 시 테이블 500개 기준 2주 소요, 휴먼 에러 빈번.
타겟: DBA, 백엔드 개발자. Windows 환경에서 GUI로 조작."

✅ Good (API 서버):
"사내 시스템 간 데이터 동기화를 중개하는 REST API 서버.
Pain Point: 시스템마다 연동 코드를 별도 작성 → 유지보수 비용 급증.
타겟: 내부 개발팀(API 소비자), DevOps(운영)."

✅ Good (웹 앱):
"소규모 팀(5~30명)이 프로젝트 진행 상황을 추적하는 웹 대시보드.
Pain Point: 스프레드시트로 관리 → 실시간 현황 파악 불가, 버전 충돌.
타겟: 프로젝트 매니저(관리), 팀원(상태 업데이트)."
```

### 2. 사용 시나리오 (Use Case)
> **PRD에서 가장 중요한 섹션.** 기능 나열보다 사용자의 행동 흐름이 먼저.

프로젝트 유형에 맞는 형태로 작성:

| 프로젝트 유형 | 시나리오 형태 |
|-------------|-------------|
| GUI 앱 (웹/모바일/데스크톱) | 화면별 사용자 행동 Step-by-Step |
| CLI 도구 | 명령어 실행 흐름 + 입출력 예시 |
| API / 라이브러리 | 호출 시나리오 + 요청/응답 예시 |
| 배치/자동화 | 트리거 → 처리 → 결과 흐름 |

**Good — GUI 앱 시나리오 (분기/예외 포함):**
```
시나리오: 데이터 변환 실행
1. 사용자가 소스 DB 접속 정보를 입력한다
2. "연결 테스트" 버튼으로 접속을 확인한다
3. [실패 시] 에러 메시지 표시, 접속 정보 재입력 유도
4. 변환 대상 테이블을 선택한다 (전체 선택 / 개별 선택)
5. 변환 옵션을 설정한다 (타입 매핑 규칙, 인덱스 포함 여부 등)
6. "변환 시작" 버튼을 누른다
7. 진행률 표시 (테이블별 진행 상태)
8. [변환 실패 테이블] 에러 로그 + 건너뛰기/중단 선택
9. 완료 시 결과 리포트 표시 (성공/실패/경고 건수)
```

**Good — API 시나리오 (요청/응답 포함):**
```
시나리오: 외부 시스템이 주문 데이터를 동기화
1. 외부 시스템이 POST /api/v1/orders 호출 (JSON body: 주문 목록)
2. [인증 실패] 401 반환 + 에러 코드
3. [입력 검증 실패] 400 반환 + 필드별 에러 상세
4. 중복 주문 ID 검출 → 기존 데이터 업데이트 (upsert)
5. 처리 완료 → 200 반환 (처리 건수, 실패 건수, 실패 상세)
6. [부분 실패] 207 반환 + 건별 성공/실패 상태
```

**Good — CLI 시나리오 (입출력 포함):**
```
시나리오: 로그 파일 분석
1. 사용자가 `logtool analyze --input ./logs/ --format json` 실행
2. [경로 없음] "Error: ./logs/ not found" + 종료 코드 1
3. 로그 파일 스캔 → 파일 수/총 크기 표시
4. 분석 진행 (프로그레스 바)
5. [Ctrl+C] 중단 확인 → 중간 결과 저장 여부 질문
6. 완료 시 요약 리포트 stdout 출력 + 상세 결과 파일 생성
```

**Bad — 기능만 나열 (어떤 유형이든):**
```
- DB 접속 기능
- 테이블 변환 기능
- 로그 기능
→ 이건 "시나리오"가 아니라 "기능 목록"이다. 사용자의 행동 흐름이 보이지 않는다.
```

**시나리오 작성 체크포인트:**
- [ ] 각 단계에서 사용자가 **무엇을 하고, 무엇을 보는지** 명확한가?
- [ ] **실패/예외 분기**가 표시되어 있는가?
- [ ] **시작점과 끝점**이 명확한가?
- [ ] **역할별로 다른 흐름**이 구분되어 있는가? (해당 시)

### 3. 기능 상세
각 기능을 아래 구조로 작성:
- **기능 설명**: 무엇을 하는 기능인지
- **사용자/호출자**: 누가 사용하는지 (역할, 권한, 외부 시스템 등)
- **입력 → 처리 → 출력**: 데이터 흐름
- **비즈니스 규칙**: 상세 로직 (상태 전이, 자동 처리, 제한 조건, 유효성 검증 등)
- **엣지 케이스**: 예외 상황과 처리 방법
- **MVP 우선순위**: Must-have / Should-have / Nice-to-have

**Good vs Bad — 비즈니스 규칙:**

```
❌ Bad:
"데이터를 처리할 수 있다." / "사용자가 관리할 수 있다."
→ 무슨 데이터? 어떤 규칙으로? 실패하면? 누가 어떤 권한으로?

✅ Good (변환 규칙):
타입 매핑 규칙:
- Oracle NUMBER(p,s) → PostgreSQL NUMERIC(p,s)
- Oracle VARCHAR2(n) → PostgreSQL VARCHAR(n)
- 매핑 불가 타입 → WARNING 로그 + 수동 변환 대상으로 표시
- 사용자 커스텀 매핑 규칙 추가 가능 (설정 파일)

✅ Good (권한 규칙):
접근 제어:
- viewer: 읽기 전용 (목록 조회, 상세 조회)
- editor: viewer + 생성/수정 (본인이 만든 항목만)
- admin: editor + 삭제 + 다른 사용자의 항목 수정
- owner: admin + 멤버 관리 + 설정 변경

✅ Good (처리량 제한):
API Rate Limit:
- 인증된 사용자: 100 req/min
- 미인증: 10 req/min
- 초과 시: 429 반환 + Retry-After 헤더
- 배치 API: 건당 최대 500건, 요청 크기 10MB 이하
```

**Good vs Bad — 상태 전이 (해당 시):**

```
❌ Bad: "작업은 여러 상태를 가진다."

✅ Good:
작업 상태 전이:
  idle → running (사용자가 "시작")
  running → completed (전체 성공)
  running → partial (일부 실패)
  running → failed (치명적 에러)
  running → cancelled (사용자 취소)
  partial/failed → running (재시도, 실패 건만 재실행)
```

> 상태 전이가 있는 기능이라면 **모든 상태 + 전이 조건 + 전이 주체**를 빠짐없이 적는다.
> "이 상태에서 저 상태로 갈 수 있나?"가 불분명하면 구현 중 반드시 혼란이 생긴다.

### 4. 데이터 모델 / 데이터 구조
> 프로젝트 유형에 따라 DB 테이블, 파일 포맷, API 스키마, 설정 파일 구조 등.

작성할 내용:
- 엔티티/테이블/스키마 목록과 설명
- 필드 목록 (이름, 타입, 필수 여부, 기본값, 설명)
- 관계/참조
- 제약 조건 (유니크, 범위, 포맷 등)
- 확장 고려사항

**Good vs Bad:**

```
❌ Bad:
"users 테이블과 projects 테이블이 있다."
→ 필드가 없으면 구현할 수 없다.

✅ Good (DB 테이블):
### projects
| 필드 | 타입 | 필수 | 기본값 | 설명 |
|------|------|------|--------|------|
| id | UUID | Y | auto | PK |
| name | TEXT | Y | - | 프로젝트명 (2~50자) |
| status | TEXT | Y | 'active' | active / archived / deleted |
| owner_id | UUID | Y | - | FK → users.id |
| created_at | TIMESTAMP | Y | NOW() | |

✅ Good (설정 파일 구조):
### config.json
{
  "source": { "type": "oracle", "host": "...", "port": 1521 },
  "target": { "type": "postgresql", "host": "...", "port": 5432 },
  "options": {
    "include_indexes": true,    // 인덱스 변환 포함 (기본: true)
    "batch_size": 1000,         // 배치 처리 단위 (기본: 1000, 범위: 100~10000)
    "on_error": "skip"          // skip: 건너뛰기, abort: 중단 (기본: skip)
  }
}

✅ Good (API 스키마):
### POST /api/v1/tasks
Request: { "title": string(필수, 2~100자), "assignee_id": UUID(선택), "priority": "low"|"medium"|"high"(기본: medium) }
Response 201: { "id": UUID, "title": string, "status": "open", "created_at": ISO8601 }
Response 400: { "error": { "code": "VALIDATION_ERROR", "fields": [...] } }
```

**자주 빠뜨리는 것:**
- status/type 등 **필터/분류 필드의 기본값** — 기본값이 없으면 예상치 못한 동작
- **삭제 전략** — 물리 삭제 vs 논리 삭제(soft delete) vs 보관(archive)
- **참조 무결성** — 부모 삭제 시 자식 데이터 처리 방법
- **입출력 포맷** — 파일 기반이면 입출력 포맷, API면 요청/응답 스키마
- **제약 조건** — 문자열 길이, 숫자 범위, 유니크 제약, 포맷(이메일, URL 등)

### 5. 알림/이벤트/로그 설계 (해당 시)
시스템에서 발생하는 이벤트와 그에 대한 반응:

| 항목 | 설명 |
|------|------|
| 트리거 | 무엇이 발생하면 |
| 대상/출력 | 누구에게 알리거나, 어디에 기록하거나 |
| 내용 | 무슨 정보를 포함하는지 |
| 행동 | 수신자가 취할 수 있는 다음 행동 |
| 우선순위 | 즉시 / 배치 / 비동기 |

> GUI 앱이면 알림/토스트, API면 웹훅/이벤트, 배치면 로그/리포트 등 프로젝트에 맞게 적용.

### 6. 우선순위 분류 (MoSCoW)
- **Must-have**: 이거 없으면 MVP가 아님
- **Should-have**: 있으면 좋지만 없어도 출시/사용 가능
- **Nice-to-have**: 여유 있으면 추가
- **Won't-have (이번에)**: 명시적으로 안 하는 것

**Won't-have가 중요한 이유:** 적지 않으면 에이전트가 범위를 잘못 판단하여 불필요한 기능을 Sprint에 넣는다.

### 7. 엣지 케이스 & 예외 처리
"이 기능이 정상적으로 동작하지 않는 모든 경우"를 나열한다. 각 케이스에 **처리 방법**을 반드시 함께 적는다.

```
❌ Bad: "에러가 발생할 수 있다"
✅ Good: "DB 연결 타임아웃(30초) → 재시도 3회 → 실패 시 에러 다이얼로그 + 로그 기록"
```

**프로젝트 유형별 공통 엣지 케이스:**

| 유형 | 자주 발생하는 엣지 케이스 |
|------|----------------------|
| 모든 유형 | 잘못된 입력, 권한 부족, 리소스 부족(메모리/디스크), 의존 서비스 장애 |
| GUI 앱 | 동시 조작, 긴 작업 중 UI 응답성, 창 닫기 중 미저장 데이터 |
| API/서버 | 동시 요청(race condition), 인증 만료, rate limit, 대용량 페이로드 |
| 파일/데이터 처리 | 손상된 입력, 인코딩 불일치, 0건/대용량, 중간 실패 시 복구 |
| CLI | 잘못된 인자, 파이프 입력, 시그널(Ctrl+C) 처리, 출력 리다이렉트 |

### 8. 장기 비전 (Phase별)
- Phase 1 (MVP): 핵심 기능만
- Phase 2~N: 확장 기능
- 각 Phase의 **핵심 가치와 의존성**

Phase 간 의존성을 명시하는 것이 중요:
```
❌ Bad:
Phase 1: 기본 기능
Phase 2: 고급 기능
Phase 3: 추가 기능
→ 각 Phase에 뭐가 들어가는지, 왜 이 순서인지 알 수 없다.

✅ Good:
Phase 1 (MVP): 인증 + 핵심 CRUD + 기본 UI
Phase 2: 검색/필터 + 알림 (Phase 1의 데이터 구조 필요)
Phase 3: 분석/리포트 (Phase 2의 데이터 축적 필요)
Phase 4: 외부 연동 API (Phase 1~3 안정화 후)
```

### 9. 비기능적 요구사항
- 성능 (처리 속도, 응답 시간, 처리량)
- 확장성 (데이터 규모, 동시 사용)
- 보안 (인증, 접근 제어, 데이터 보호)
- 호환성 (OS, 런타임 버전, 브라우저 등)
- 안정성 (에러 복구, 로깅, 모니터링)

**구체적 기준이 없으면 무의미:**

```
❌ Bad: "빨라야 한다" / "안전해야 한다" / "확장 가능해야 한다"

✅ Good (성능):
- API 응답: p95 기준 200ms 이내
- 파일 처리: 100MB 파일 기준 30초 이내
- 동시 접속: 100명 동시 사용 시 성능 저하 없음

✅ Good (보안):
- 인증: OAuth 2.0 또는 API Key
- 민감 데이터: AES-256 암호화 저장
- 로그: 개인정보 마스킹 필수

✅ Good (호환성):
- OS: Windows 10 이상
- 브라우저: Chrome/Edge 최근 2개 버전
- DB: PostgreSQL 14+, MySQL 8+
```

### 10. 경쟁 분석 / 기존 도구 비교 (선택)
- 유사 도구/서비스와 차별점
- 참고할 UX/기능

### 11. 성공 지표 (KPI)
- 정량적 측정 기준
- MVP 성공 판단 기준

프로젝트 유형별 KPI 예시:

| 유형 | KPI 예시 |
|------|---------|
| 사용자 서비스 (웹/앱) | DAU, 가입 전환율, 주간 리텐션, 핵심 기능 사용률 |
| 내부 도구 / 데스크톱 | 작업 완료 시간 단축률, 에러 발생률, 사용자 만족도 |
| API / 라이브러리 | 호출 성공률, 평균 응답 시간, 도입 팀/서비스 수 |
| 배치 / 자동화 | 처리 성공률, 실행 시간, 수동 개입 빈도 |

## PRD 작성 안티패턴

| 안티패턴 | 결과 | 대안 |
|---------|------|------|
| **기능만 나열하고 시나리오 없음** | 에이전트가 기능 간 연결/순서를 모름 → Sprint 순서가 엉뚱함 | 사용 시나리오를 먼저 작성하고, 기능은 시나리오에서 도출 |
| **"~할 수 있다" 식 애매한 표현** | 구체적 조건/제약이 없어 구현 시 매번 질문 | 주어 + 조건 + 동작 + 결과를 명시 |
| **우선순위 미분류** | 에이전트가 모든 기능을 Phase 1에 넣음 → MVP가 비대 | MoSCoW 분류 필수, Won't-have도 명시 |
| **엣지 케이스 없음/대충** | 구현 후 "이 경우는 어떻게 하죠?" 반복 | 최소 5개 이상 예외 상황 + 처리 방법 |
| **데이터 구조가 이름만** | 에이전트가 필드를 추측하여 설계 → 나중에 구조 변경 | 필드, 타입, 기본값, 제약 조건을 모두 명시 |
| **상태 전이 미정의** | 구현자마다 상태를 다르게 해석 → 버그 | 상태 다이어그램 또는 전이 표 작성 |
| **"나중에 정하자" / "추후 논의"** | 에이전트가 그 부분을 건너뛰거나 임의 결정 | 미정이면 Won't-have에 넣거나, 최소 방향이라도 명시 |
| **비기능 요구사항이 "잘 해야 한다" 수준** | 측정 기준이 없어 검증 불가 | 구체적 숫자와 기준 제시 |
| **"등등", "기타", "필요에 따라"** | 에이전트가 범위를 해석 불가 | 구체적으로 나열하거나 Won't-have로 분류 |

## 섹션별 "충분한 상세도" 판단 기준

| 섹션 | 부족한 상태 | 충분한 상태 |
|------|-----------|-----------|
| 제품 개요 | 한 줄 설명만 | Pain Point + 타겟 사용자 + 제품 유형 명확 |
| 사용 시나리오 | 기능 목록 나열 | Step-by-Step, 분기/예외 포함, 주요 시나리오 3개+ |
| 기능 상세 | "CRUD 가능" | 입력→처리→출력, 비즈니스 규칙, 상태 전이, 엣지 케이스 |
| 데이터 구조 | 이름만 | 필드 + 타입 + 기본값 + 제약 조건 + 관계 |
| 엣지 케이스 | 0~2개 또는 처리 방법 없음 | 5개+, 각각에 **구체적 처리 방법** 포함 |
| 비기능 요구사항 | "빠르게, 안전하게" | 숫자 기준 (시간, 크기, 동시 수 등) |
| 우선순위 | 전부 Must-have | MoSCoW 4단계 분류 + Won't-have 포함 |

## prd-to-roadmap 에이전트 연동

에이전트가 PRD를 파싱하여 ROADMAP을 자동 생성할 때, 아래 조건이 갖춰지면 고품질 결과가 나온다.

### 에이전트가 필요로 하는 정보

| 정보 | 용도 |
|------|------|
| 기능별 MoSCoW 태그 | Phase 배치 결정 |
| 기능 간 의존성 | Phase 순서 결정 |
| 사용 시나리오 | Sprint 단위 분할 기준 |
| 데이터 구조 | 기반 인프라 Sprint 도출 |
| 비기능 요구사항 | 성능/보안 Sprint 도출 |

### 에이전트가 해석 못 하는 표현

| 표현 | 문제 | 수정 방법 |
|------|------|----------|
| "적절히 처리한다" | 무엇을 어떻게인지 불명 | 구체적 처리 방법 명시 |
| "필요에 따라 추가" | 조건이 불명 | 조건을 명시하거나 Won't-have로 |
| "추후 논의" | 에이전트가 건너뜀 | 최소 방향이라도 명시 |
| "등등", "기타" | 범위 불명 | 구체적 나열 |
| 우선순위 없는 기능 나열 | 전부 Phase 1에 투입 | MoSCoW 태그 필수 |

## 품질 체크리스트

### 기본 (70점)
- [ ] 필수 섹션(1~9)이 채워졌는가? (해당 없음이면 사유 명시)
- [ ] 타겟 사용자/소비자가 정의되었는가?
- [ ] MVP 범위가 명확한가? (Must-have 목록)
- [ ] Won't-have가 명시되어 있는가?
- [ ] 프로젝트 유형(웹/앱/CLI/API 등)이 명확한가?

### 시니어 (85점)
- [ ] **사용 시나리오**가 Step-by-Step으로 작성되었는가? (기능 나열 X)
- [ ] 시나리오에 **분기/예외 흐름**이 포함되었는가?
- [ ] **비즈니스 규칙**에 구체적 조건/숫자가 있는가?
- [ ] **엣지 케이스** 5개 이상, 각각에 **처리 방법** 포함?
- [ ] **MoSCoW 우선순위** 4단계 분류 완료?
- [ ] **데이터 구조**에 필드/타입/기본값/제약 조건 포함?
- [ ] **"~할 수 있다" 식 애매한 표현**이 없는가?
- [ ] 안티패턴 테이블의 항목을 하나도 위반하지 않는가?

### 최고 (90점+)
- [ ] **성공 지표(KPI)**가 정량적으로 정의되었는가?
- [ ] **이벤트/알림/로그 설계**가 테이블로 정리되었는가? (해당 시)
- [ ] **경쟁 분석** 또는 기존 도구 대비 차별점이 포함되었는가?
- [ ] Phase별 장기 비전에 **핵심 가치와 의존성**이 있는가?
- [ ] **비기능 요구사항**에 구체적 숫자가 있는가?
- [ ] prd-to-roadmap 에이전트가 **이 PRD만 읽고** ROADMAP을 만들 수 있는가?
