# rules/{tech}.md 작성 가이드

> 이 파일은 기술 스택 규칙 파일을 작성할 때 참조하는 **메타 템플릿**입니다.
> 새로운 rules/{tech}.md를 생성할 때 이 구조와 기준을 따르면, 모든 에이전트가
> 해당 기술의 **시니어 개발자 수준**으로 동작하고, 바이브 코딩 시 **최고 품질의 결과물**을 보장합니다.

---

## 작성 워크플로우 (필수)

> rules 파일은 한 번에 쓰고 끝내는 게 아니다.
> **3단계 작성 → 자가 검증 → 점진적 개선** 사이클을 따른다.

### Step 1: 기술 고유 영역 발굴 (작성 전)

아래 템플릿의 **공통 필수 섹션**은 모든 기술에 적용되는 최소 골격이다.
하지만 모든 기술에는 **그 기술만의 고유 영역**이 있다.

작성 전에 반드시 다음 질문에 답한다:

| 질문 | 목적 |
|------|------|
| 이 기술에서 **초보가 가장 많이 실수하는 영역** 3~5개는? | 안티패턴 + 실전 함정 발굴 |
| 이 기술의 **플랫폼/환경별 차이**가 있는가? | 호환성 규칙 도출 |
| 이 기술에서 **성능 병목이 자주 발생하는 지점**은? | 성능 섹션 구체화 |
| 이 기술의 **보안 취약점 Top 5**는? | 보안 섹션 구체화 |
| 공통 필수 섹션 외에 **이 기술에만 필요한 고유 섹션**은? | 기술 고유 섹션 추가 |

**기술 고유 섹션 예시** (이것은 예시일 뿐, 기술마다 다르다):
- 프론트엔드 UI: 애니메이션, 키보드 처리, 접근성, 반응형, 상태 관리 심화
- 백엔드 API: 인증/인가 패턴, Rate Limiting, 데이터베이스 쿼리 최적화, 마이그레이션
- 데이터베이스: 인덱싱 전략, 접근 제어 정책, 트랜잭션 패턴, 마이그레이션
- 모바일: 앱 생명주기, 푸시 알림, 딥링크, 오프라인 대응

발굴한 기술 고유 섹션을 공통 필수 섹션에 **추가하여** 목차를 완성한 후 작성을 시작한다.

### Step 2: 작성 + 중간 점검 + 자가 검증

#### 2-1. 작성 순서 (카테고리 의식 작성)

Step 1에서 완성한 목차를 기반으로 작성하되, **어떤 카테고리에 기여하는 섹션인지** 의식하며 쓴다:

```
① 구조 섹션 (핵심원칙 → 프로젝트구조 → 네이밍 → 포맷팅 → import → 파일순서)
   → 중간 점검: A(구조) 6점+ 도달?

② 패턴 섹션 (코드구조&패턴 → Null처리 → 비동기 → 에러처리)
   → 중간 점검: B(코드예시) 11점+ 도달? Good/Bad 쌍 10개+?

③ 방어 섹션 (안티패턴 → 보안 → 성능 → 실전함정)
   → 중간 점검: C(안티패턴) 9점+? D(함정) 11점+?

④ 검증 섹션 (테스트 → 검증체크리스트 → 우회금지 → 문서화)
   → 중간 점검: E(검증) 9점+?

⑤ 설정 섹션 (빌드&설정 → 라이브러리 → 확장포인트)
```

> 이 순서를 따르면 섹션 그룹마다 중간 점검이 가능하고, 부족한 카테고리를 조기에 발견한다.

#### 2-2. 프로젝트 종속 검증 (🔴 필수 — 건너뛸 수 없음)

작성 완료 후 **반드시** 파일 전체에서 프로젝트 도메인 용어를 검색한다:
- `grep -i` 등으로 프로젝트 테이블명, 도메인 용어, 프로젝트 고유명사를 검색
- 1건이라도 발견되면 범용 용어로 교체한 후 다음 단계로 진행
- **범용 예시 기준**: `teams`, `team_members`, `tasks`, `profiles`, `categories`
- **금지 예시**: 프로젝트 DB 테이블명, 프로젝트 고유 도메인 용어, 프로젝트명

#### 2-3. 자가 채점

아래 **품질 평가 체계**의 자가 채점 워크시트를 사용하여 6개 카테고리별 채점.
**각 카테고리에 정량 근거를 기입** — "코드 블록 25개, Good/Bad 쌍 12개" 식으로.

#### 2-4. 배포 판정

- **총점 60점 미만 → 배포 금지** (Level 2 미달)
- **카테고리 최소 요건 미충족 → 해당 카테고리 보강 후 재채점**

> **첫 작성 최소 기준은 Level 2 충족(총점 60점 + 카테고리 최소 요건)이다.**
> Level 1(형식만 갖춤)은 총점 40~59점이며, 이 상태로 코드 작성을 시작하면 실전에서 버그가 터진다.
>
> **자가 검증 시 핵심 질문**: "이 규칙만 읽고 AI가 코드를 생성했을 때, 시니어가 작성한 것처럼 보이는가?"
> 이 질문에 "아니오"면 Level 2 미달이다.

### Step 3: 점진적 개선 (작업하면서 발전)

rules 파일은 **살아있는 문서**다. 작업 중 아래 상황이 발생하면 AI가 **능동적으로** 규칙을 업데이트한다:

| 상황 | 행동 |
|------|------|
| 작업 중 규칙에 없는 함정/버그를 발견 | 해당 섹션에 안티패턴 또는 주의사항 추가 |
| 사용자가 코드 품질을 지적 | 원인을 분석하고 재발 방지 규칙 추가 |
| 같은 실수를 2번 이상 반복 | 규칙이 부족한 것 — 구체적 코드 예시와 함께 보강 |
| 새로운 고급 패턴을 적용 | 해당 패턴을 규칙에 추가 (다음에도 적용하기 위해) |
| 라이브러리/프레임워크 버전 업데이트 | 변경된 권장사항 반영 |

> **핵심 원칙: 사용자가 모든 기술을 알고 있는 것이 아니다.**
> AI가 해당 기술의 전문가로서 문제를 발견하면 사용자에게 알리고,
> 규칙을 개선하여 같은 문제가 반복되지 않게 한다.

## 작성 원칙

> 각 원칙 뒤의 `[→ 카테고리 X]`는 이 원칙이 품질 평가 체계의 어느 카테고리에 반영되는지 표시.
> 작성 시 이 연결을 의식하면 자연스럽게 높은 점수가 나온다.

### 1. 바이브 코딩 최적화 `[→ 카테고리 F: 20점]`

rules 파일의 **궁극적 목적**은 AI가 읽고 시니어 수준 코드를 생성하는 것이다.
모든 규칙은 이 질문을 통과해야 한다: **"AI가 이것만 읽고 바로 좋은 코드를 생성할 수 있는가?"**

- **모호한 규칙 금지** — "적절한 에러 처리" ❌ → "UI 경계에서만 try-catch, 내부는 예외 전파, catch 시 `errors.Add($"{context}: {ex.Message}")`" ✅
- **코드 예시 = 복사 → 변수명 교체 → 바로 사용** 수준이어야 한다. 설명만 있으면 AI는 자기 방식으로 생성한다.
- **Good/Bad 쌍 필수** — Good만 보여주면 AI는 Bad를 인식하지 못해 안티패턴을 생성한다.
- **판단 기준 명시** — "이 상황이면 A 패턴, 저 상황이면 B 패턴" 선택 흐름이 있어야 AI가 상황에 맞는 코드를 생성한다.
- **네이밍 규칙은 테이블로** — AI가 주관적 판단 없이 기계적으로 적용 가능해야 한다.

### 2. 시니어 수준 전문성 `[→ 카테고리 C: 15점, D: 20점]`

공식 문서 내용만 정리하면 **Level 1(40~59점)**이다. 실전에서 버그를 방지하려면:

- **안티패턴**: 최소 **10개+**, 테이블 형식 `| 안티패턴 | 실패 결과 | 대안 |`
  - "실패 결과"는 구체적 시나리오: "비효율적이다" ❌ → "N+1 쿼리로 응답 10초 초과" ✅
  - 3분류: 일반 안티패턴 + **AI가 흔히 생성하는 실수** + 실전에서 발견된 지뢰
- **실전 함정**: 최소 **6개+**, 4요소 완비 `| 증상 | 원인 | 해결 | 오답 |`
  - 함정 = "정상으로 보이지만 특정 조건에서 깨지는 것" (안티패턴보다 위험)
  - **오답 컬럼 필수** — 없으면 AI가 땜빵 해결책을 선택한다
- **고급 패턴**: "언제 쓰고 언제 쓰지 않는지" 판단 기준과 함께 제시
- **"왜"를 항상 명시**: 모든 규칙에 "이렇게 안 하면 뭐가 깨지는지" 한 줄

### 3. 정량 기준 충족 `[→ 카테고리 A: 10점, B: 20점]`

| 항목 | 최소 기준 (Level 2) | 권장 기준 (Level 3) |
|------|:------------------:|:------------------:|
| 코드 블록 | 20개+ | 30개+ |
| Good/Bad 쌍 | 10개+ | 15개+ |
| 안티패턴 테이블 항목 | 10개+ | 15개+ |
| 실전 함정 (4요소 완비) | 3개+ | 7개+ |
| 검증 체크리스트 | 2개+ | 5개+ |
| 네이밍 컨벤션 테이블 행 | 8개+ | 12개+ |

> 이 표를 작성 중에 참조한다. 각 섹션을 쓸 때 "이 섹션에서 코드 블록 몇 개 나왔지?" 체크.

### 4. 프로젝트 독립성 `[→ 상한선: 위반 시 50점]`

🔴 **가장 치명적인 결함** — 프로젝트 종속 용어가 1건이라도 있으면 **50점 상한**.

- 코드 예시의 테이블명/변수명도 포함: `clubs`, `schedule_events` ❌ → `teams`, `tasks` ✅
- **흔한 실수**: 프로젝트에서 겪은 경험을 규칙에 추가할 때 프로젝트 코드를 그대로 복사
- **경험은 반영하되 용어는 범용화**한다.
- 해당 기술 스택을 사용하는 **모든 프로젝트**에 적용 가능해야 한다.
- 프로젝트별 설정은 CLAUDE.md에 위임한다.
- **검증**: 작성/수정 후 반드시 `grep`으로 프로젝트 도메인 용어 0건 확인.

### 5. 실전 검증 필수 `[→ 카테고리 D, E]`

- 규칙에 쓴 코드 예시가 **실제 환경에서 동작하는지** 검증한다. 이론만 맞고 실전에서 깨지는 예시는 독이다.
- **"이 규칙을 따랐는데도 문제가 발생한" 경우가 곧 규칙의 결함**이다 — 즉시 보완한다.
- 최종 검증: "이 규칙만 읽고 코드를 작성했을 때, 지금까지 겪은 모든 버그가 사전에 방지되는가?"

## Level별 품질 차이 예시

> 같은 섹션이라도 Level에 따라 이렇게 다르다.
> 작성 시 "내가 쓴 것이 Level 1인지 3인지" 스스로 판별하는 기준으로 사용한다.

### 예시 1: 안티패턴 섹션

**Level 1 (40~59점) — 이렇게 쓰면 부족하다:**

```markdown
## 안티패턴
| 안티패턴 | 이유 | 대안 |
|---------|------|------|
| any 타입 사용 | 타입 안전성 위반 | unknown 사용 |
| console.log 남용 | 좋지 않다 | 로깅 라이브러리 |
| var 사용 | 호이스팅 문제 | const/let |
```
→ **문제**: "이유"가 "좋지 않다", "위반" 같은 일반론. AI가 이걸 읽어도 *왜* 안 되는지 모르고 변형 생성.

**Level 3 (75~89점) — 이렇게 써야 한다:**

```markdown
## 안티패턴 (하지 말 것)

### 일반 안티패턴
| 안티패턴 | 실패 결과 | 대안 |
|---------|----------|------|
| `any` 타입 | 타입 시스템 무력화 → 런타임 `Cannot read property of undefined` 크래시 | `unknown` + 타입 가드 |
| `console.log` 프로덕션 | 민감 데이터(토큰, 개인정보) 브라우저 콘솔 노출 | `Logger.debug()` + 환경별 제어 |
| 문자열 `+` 루프 | 100개 연결 시 100개 중간 string 생성 → GC 압박, O(N²) | `StringBuilder` 또는 템플릿 리터럴 |

### AI가 흔히 생성하는 실수
| 안티패턴 | 실패 결과 | 대안 |
|---------|----------|------|
| 모든 메서드에 `try-catch` | 에러 전파 차단 → 상위 레이어가 문제 모름 → 빈 화면 | 경계(UI 핸들러)에서만 catch |
| `?.` 과 `!` 남발 | `!`가 틀리면 NullReferenceException → 프로덕션 크래시 | 적절한 null 검증 + early return |
```
→ **차이**: 실패 결과가 구체적 시나리오, AI 실수 분류 별도, 모든 항목에 대안 코드

### 예시 2: 에러 처리 섹션

**Level 1:**

```markdown
## 에러 처리
- try-catch를 적절히 사용한다
- 에러 메시지를 로깅한다
- 사용자에게 알려준다
```
→ **문제**: "적절히"가 뭔지 모호. AI가 모든 메서드에 try-catch를 감싸는 코드 생성.

**Level 3:**

```markdown
## 에러/예외 처리

### 경계에서만 catch — 내부는 전파
​```csharp
// ✅ UI 이벤트 핸들러 (경계) — 여기서 catch
private async void btnAnalyze_Click(object? sender, EventArgs e)
{
    try
    {
        var result = await AnalyzeAsync(path);
        UpdateUI(result);
    }
    catch (Exception ex)
    {
        SetStatus($"분석 실패: {ex.Message}");
    }
}

// ✅ 내부 메서드 — catch 하지 않고 전파
public AnalysisResult Analyze(IReadOnlyList<string> files)
{
    // 예외가 발생하면 호출자(UI)로 전파
    var tree = CSharpSyntaxTree.ParseText(code);
    ...
}

// ❌ 내부에서도 catch — 에러가 묻히고 상위에서 문제 인지 못함
public AnalysisResult Analyze(IReadOnlyList<string> files)
{
    try { ... }
    catch (Exception ex) { return new AnalysisResult(); }  // 빈 결과 반환 — 왜 빈지 모름
}
​```

### 파일별 부분 실패 허용 패턴
​```csharp
// ✅ 파일 하나 실패해도 나머지 계속 — Errors 리스트로 추적
foreach (var file in files)
{
    try { ... }
    catch (Exception ex)
    {
        result.Errors.Add($"{Path.GetFileName(file)}: {ex.Message}");
    }
}
​```
```
→ **차이**: "어디서 catch하고 어디서 안 하는지" 명확한 기준 + 복사 가능한 패턴 + Good/Bad 대비

### 예시 3: 검증 체크리스트

**Level 1 (없음):**
→ 아예 섹션이 없거나 "테스트를 작성한다" 수준

**Level 3:**

```markdown
### 새 기능 추가 시
- [ ] 빌드 성공 (`dotnet build` — 0 errors, 0 warnings)
- [ ] 기존 테스트 전원 통과 (`dotnet test`)
- [ ] 새 기능에 대한 테스트 케이스 추가

#### F. 바이브 코딩 유효성 (20점)

> **가장 중요한 카테고리** — rules 파일의 궁극적 목적은

| 점수 | 기준 |
|:----:|------|
| 0~5 | 규칙이 선언적 — "좋은 코드를 작성하라" 수준. AI가 읽어도 행동 변화 없음 |

| 11~14 | 패턴 + 판단 기준("이 상황이면 A, 저 상황이면 B") + Good/Bad 쌍 |
| 15~17 | 위 + 네이밍/구조/에러처리 규칙이 기계적 적용 가능 (모호하지 않음) |
| 18~20 | 위 + AI 생성 코드가 시니어 리뷰를 통과하는 수준 확인됨 |

**F 카테고리 5대 검증 질문 (하나라도 "아니오"면 14점 이하):**

1. **복사 적용성**: 코드 예시를 복사하고 변수명만 바꾸면 바로 쓸 수 있는가?
2. **네이밍 기계성**: 네이밍 규칙이 테이블로 정리되어 주관적 판단 없이 적용 가능한가?
3. **의사결정 흐름**: "이 상황에서 어떤 패턴을 쓸지" 선택 기준이 명시되었는가?
4. **AI 실수 사전 차단**: AI가 흔히 생성하는 실수가 안티패턴에 명시되어 있는가?
5. **판단 영역 구체화**: 에러 처리/null 처리 등 판단이 필요한 영역에 명확한 기준이 있는가?

**"기계적 적용 가능" 예시:**

```

"적절한 에러 처리를 합니다"
→ "적절한"이 뭔지 AI마다 다르게 해석

✅ 기계적 규칙 (AI가 바로 적용):
"UI 이벤트 핸들러 경계에서만 try-catch. 내부 메서드는 예외 전파.
 catch 시 result.Errors.Add($"{context}: {ex.Message}") 패턴."
→ 어디서, 어떻게, 어떤 형식으로 — 모호함 없음
```

#### 카테고리 최소 요건 (Level Gate)

총점이 범위 안에 있어도 **카테고리 최소 요건**을 하나라도 미충족하면 **하위 Level**로 강제 판정.
이 규칙은 "한 카테고리만 극단적으로 높이고 나머지를 방치"하는 편식을 방지한다.

| Level | 최소 요건 |
|:-----:|:--------:|
| 2 | 모든 카테고리에서 배점의 **35%** 이상 |
| 3 | 모든 카테고리에서 배점의 **55%** 이상 |
| 4 | 모든 카테고리에서 배점의 **75%** 이상 |

**카테고리별 최소 점수 (구체적 수치):**

| Level | A (10) | B (20) | C (15) | D (20) | E (15) | F (20) |
|:-----:|:------:|:------:|:------:|:------:|:------:|:------:|
| 2 | ≥4 | ≥7 | ≥6 | ≥7 | ≥6 | ≥7 |
| 3 | ≥6 | ≥11 | ≥9 | ≥11 | ≥9 | ≥11 |
| 4 | ≥8 | ≥15 | ≥12 | ≥15 | ≥12 | ≥15 |

```

### 자가 채점 워크시트

rules 파일 작성/개선 후 아래 워크시트를 복사하여 채점한다.
**각 카테고리에 "근거"를 반드시 기입** — 근거 없는 점수는 편향된 자기 평가.

```
## 자가 채점 결과: {tech}.md

| 카테고리 | 배점 | 점수 | 근거 (정량 수치 필수) |
|---------|:----:|:----:|---------------------|
| A. 구조 완성도 | 10 | /10 | 필수 섹션 _개, 고유 섹션 _개 |
| B. 코드 예시 품질 | 20 | /20 | 코드 블록 _개, Good/Bad 쌍 _개 |
| C. 안티패턴 구체성 | 15 | /15 | 항목 _개, 구체적 실패 결과 _%, 분류 _개 |
| D. 실전 함정 깊이 | 20 | /20 | 함정 _개, 4요소 완비 _개 |
| E. 검증 가능성 | 15 | /15 | 체크리스트 _개, 디버깅 흐름 _개 |
| F. 바이브 코딩 유효성 | 20 | /20 | 5대 질문 중 "예" _개 |
| **합계** | **100** | **/100** | |

### 상한선 체크
- [ ] 프로젝트 종속 용어 0건 확인 (grep 실행)
- [ ] 필수 섹션 모두 존재
- [ ] 모든 필수 섹션에 코드 예시 1개+
- [ ] 안티패턴에 실패 결과 명시

상한선 적용: ___점 (해당 없으면 "없음")
카테고리 최소 요건 충족: Level ___

**최종 점수**: min(합계, 상한선) = ___점
**최종 Level**: min(점수 기반 Level, 카테고리 Gate Level) = Level ___
```

### 채점 후 처리

- rules 파일을 신규 작성하거나 대폭 수정한 후 자가 채점 실행 (선택).
- 채점 결과는 사용자에게 보고 — 외부 문서(README 등) 갱신 여부는 사용자가 판단.

## 작성 순서 권장

여러 기술 스택의 rules를 생성할 때:

1. **언어 규칙 먼저** (예: `typescript.md`, `python.md`)
2. **프레임워크 규칙** (예: `react-native.md`, `fastapi.md`)
3. **인프라/서비스 규칙** (예: `supabase.md`, `firebase.md`)

언어 규칙이 프레임워크의 기반이 되므로, 이 순서를 따르면 일관성이 유지됩니다.
프레임워크 규칙에서는 언어 규칙과 중복되는 내용을 반복하지 않고 "typescript.md 참조"로 위임합니다.

## 레퍼런스 품질 기준

완성된 rules 파일은 아래와 동등한 수준이어야 합니다:
- Google Style Guide (해당 언어 버전) — 형식/컨벤션 참고
- Airbnb JavaScript/React Style Guide — 규칙의 구체성 참고
- 해당 기술 공식 문서의 Best Practices 섹션 — 기본 내용 커버 확인
- **시니어 개발자가 코드 리뷰에서 지적하는 수준의 규칙** — 핵심 기준
- **프로덕션 장애 포스트모템에서 나오는 교훈** — 실전 함정의 출처

**"공식 문서를 번역한 수준"은 Level 1(40~59점)입니다.**
**"실전 함정 + 구체적 실패 시나리오"가 포함되어야 Level 2(60점+)입니다.**
**"검증 체크리스트 + 바이브 코딩 유효성"까지 있어야 Level 3(75점+)입니다.**
**"이 규칙을 따랐는데도 버그가 났다"면 규칙이 부족한 것이지, 따르는 사람의 문제가 아닙니다.**

## 점진적 개선 프로세스

> rules 파일은 한 번 쓰고 끝나는 문서가 아니다.
> 작업하면서 발견한 함정, 사용자 피드백, 새로운 패턴이 축적되어 전문가 수준으로 성장한다.

### 개선 트리거 (이 상황이 발생하면 규칙을 업데이트한다)

```
작업 중 버그/함정 발견
  → "실전 함정 & 지뢰" 섹션에 증상/원인/해결/오답 추가

규칙을 따랐는데도 버그 발생
  → 규칙의 결함이다 — 해당 규칙을 즉시 보완 + 검증 체크리스트에 항목 추가
  → 이것이 가장 중요한 트리거다. 규칙이 실전을 커버하지 못하는 증거.

사용자가 코드 품질 지적
  → 원인 분석 → 재발 방지 규칙 + Good/Bad 코드 예시 추가

우회/땜빵으로 해결한 후 정석으로 재수정한 경우
  → "우회/핵 금지 원칙" 섹션에 해당 사례 추가 (왜 우회가 잘못이었는지)

같은 실수 2회 이상 반복
  → 규칙이 부족하거나 모호한 것 → 구체적 코드 예시로 보강

새로운 고급 패턴 적용 성공
  → 패턴 섹션에 추가 (다음에도 같은 수준으로 코드 생성하기 위해)

라이브러리/프레임워크 버전 변경
  → 변경된 권장사항, deprecated API 반영
```

### AI의 역할

- **사용자가 모든 기술 스택을 깊이 아는 것은 아니다**
- AI는 해당 기술의 전문가로서, 문제를 발견하면 사용자에게 알리고 규칙 개선을 제안한다
- 사용자가 "이거 왜 이래?" 라고 물을 때까지 기다리지 않는다 — **능동적으로** 개선한다
- 규칙 업데이트 시 변경 사유를 간단히 남긴다 (왜 이 규칙이 추가되었는지 이후 문맥 이해를 위해)
